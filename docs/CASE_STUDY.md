# Case Study — Building an AI-Native TB Chest X-Ray Triage System

*A first-person engineering narrative. Maintained as the project progresses; newest milestones appended at the bottom. Written so it can be lifted into a portfolio.*

---

## TL;DR

I built a frontend-only, bring-your-own-key TB chest X-ray triage app — a five-stage AI pipeline (quality gate → perception ensemble → retrieval → adjudication → verdict) wired to Hugging Face, Replicate, and OpenAI directly from the browser. Then I did the unglamorous part most demos skip: I **measured it against ground truth**, found it caught only ~14% of TB cases, and spent the rest of the project honestly closing that gap — improving the vision-language reasoning to ~42% sensitivity, building a calibration layer that sets a sensitivity-targeted operating point (with the honest caveat — surfaced by an expert panel — that its coverage is in-distribution and must be re-fit per deployment site), and standing up an offline training pipeline for a real, validated TB classifier. The throughline is intellectual honesty: at every step I reported the real numbers and the real limitations rather than the demo-friendly ones.

---

## Context & role

Solo engineer, end to end: product framing, architecture, implementation, ML evaluation, research, and the writing you're reading. The brief was a frontend-only ("no backend, no DB server") AI-native radiology triage UI with strict constraints — TypeScript strict mode, IndexedDB persistence, BYOK API keys, and provider fallback that's always *visible* to the clinician. The hard requirement underneath all of it: **it must not lie about its own confidence.** A screening tool that hides uncertainty is worse than none.

## Milestone 1 — The app (2026-05-23)

I built the full system in one pass: Vite + React 18 + TypeScript (strict, zero `any`), Tailwind + shadcn-style Radix primitives, Dexie/IndexedDB, a five-stage pipeline, a live "Agent Trace" panel, a `/validate` metrics route, a standalone single-file `demo.html`, and a README.

**Decisions I'm proud of:**
- **The normalized result contract came first.** Before any provider code, I defined `{ tb_prob, raw, provider_used, latency_ms }`. Every HF/Replicate/OpenAI payload collapses to that shape at a `parseOutput` seam, so the orchestrator and UI never see provider-specific structures. This one decision kept a three-provider system coherent.
- **Fallback is a UI event, not a hidden retry.** When Hugging Face is cold and the system falls back to Replicate, the trace card animates "→ Replicate." Degraded inference is something the clinician *sees*.
- **A deterministic safety net wraps the LLM.** The GPT adjudicator advises; hard-coded guardrails decide. A confident verdict on a degraded run is structurally impossible.

**A judgment call:** the spec said `raw: any`. Strict TypeScript bans `any`, so I used `unknown` and narrowed at the edges — documented as a deliberate deviation. Small thing, but it's the kind of decision I'd rather make explicitly than paper over.

Result: clean build, **Lighthouse accessibility 100/100** (verified with a real headless-Chrome run, not asserted).

## Milestone 2 — First contact with reality (2026-05-24)

The user dropped in real API keys, and the app met the real world — which is always messier than the spec:
- **`gpt-5.5` is real** (resolves to `gpt-5.5-2026-04-23`); the Responses API shape I'd coded matched. Quality gate, vision read, and adjudication all worked first try.
- **Hugging Face had retired `api-inference.huggingface.co`** in favor of `router.huggingface.co` — I switched the client. More importantly, on the free tier *every* community CXR model returns "not supported by provider" — serverless only hosts actively-deployed models now. A genuine 2026 constraint the spec couldn't have known.
- **Replicate sends no CORS headers**, so direct browser calls are blocked. My Node test passed (Node ignores CORS); the browser didn't. I added a Vite dev proxy and wired Replicate's CLIP model as the embedding fallback so retrieval actually works locally.

I also pulled real chest X-rays from public Hugging Face datasets to use as demo samples — verifying by eye that the "TB" ones actually showed upper-zone pathology, rather than trusting the labels.

**Lesson reinforced:** integration archaeology — finding out what the APIs *actually* do in 2026 — was as much of the work as writing code. And a test that can't reproduce the production environment (CORS) gives false confidence.

## Milestone 3 — Measuring the truth (2026-05-24)

This is the milestone I'd put first in an interview. I built a faithful Node harness mirroring the pipeline and ran it against a balanced, held-out set of 30 real labeled CXRs.

**The current system caught ~1 in 7 TB cases.** Sensitivity 14.3%, specificity 100%, AUC 0.764. As a screen — where a missed TB case is the dangerous error — that's not usable, and I said so plainly. The system was behaving as "assume everything's normal," which looks great on accuracy and fails the only metric that matters.

I resisted the urge to spin it. The value of a triage tool is bounded by its sensitivity, and 14% is a finding, not a failure to hide.

## Milestone 4 — Making the reasoner better (2026-05-24)

I researched the current literature on harnessing VLMs for radiology (chain-of-thought reading, self-consistency, threshold calibration, the overconfidence-isn't-fixed-by-prompting result) and implemented the wins:
- **Structured zone-by-zone chain-of-thought** prompt with a screening prior ("don't dismiss subtle findings").
- **Self-consistency**: K=3 reads → mean probability + spread as a *real* uncertainty estimate (verbalized confidence alone is known not to fix overconfidence).
- **Screening-biased operating point** + a safety-net combine where the model can only ever *escalate*, never clear a flagged case.

Re-measured on the same 30 images: **sensitivity 14% → 42%, AUC 0.76 → 0.84, specificity held at 100%.** A 3× gain at zero specificity cost. Still far from the WHO triage bar (≥90% sensitivity), but the remaining gap is a *perception* limit — a general vision model genuinely can't see subtle TB — not a tuning one.

## Milestone 5 — How do we actually reach 90%? (2026-05-24)

I ran the research as **parallel agent swarms** — two waves, ten focused agents total — mapping the whole solution space: SOTA TB models, in-browser ONNX inference, hosted deployment + CORS realities, datasets and training recipes, calibration/fusion/conformal methods, and CXR embedding models. The waves converged hard on one conclusion:

> There is **no off-the-shelf, externally-validated, browser-ready TB classifier.** Reaching 90% requires a real TB CNN in the primary slot; calibration converts model quality into an operating point but cannot manufacture AUC.

The sharpest insight: **at AUC ≈0.76, forcing 90% sensitivity collapses specificity toward zero.** The lever is the model, not the threshold.

---

## Milestone 6 — The calibration core, built with discipline (2026-05-24)

I wrote an implementation plan and executed it **subagent-driven**: a fresh agent per task, each gated by a spec-compliance review *then* a code-quality review before moving on. Fourteen commits. The review gates earned their keep — they caught two real bugs I'd have shipped:
- A **calibrated-vs-raw probability mismatch**: the VLM safety threshold was fit on calibrated probabilities but compared against raw ones, so the sensitivity-critical escalation would fire at the wrong point once calibration was active.
- A **degenerate evaluation split** that silently put all positives in one half.

What shipped (pure TypeScript, dependency-free, backward-compatible): temperature/Platt **probability calibration**, **log-odds fusion** of the CNN + VLM readers, and **class-conditional (Mondrian) conformal thresholds** that *guarantee* a sensitivity target by construction — all fit from a labeled holdout via a new "Calibrate" action in `/validate`, gated to fall back to the validated default policy on under-sampled fits.

A measurement here taught me something: on a tiny calibration set the conformal band correctly went maximally conservative (refer almost everything) — and the `incomplete` safeguard I'd built kicked in and reverted to the safe default. The math behaving conservatively at low N, and the guard catching it, was the system working as designed.

## Milestone 7 — Training a real perception model (in progress, 2026-05-24)

The honest path to 90% is a real, validated TB classifier. I chose a **frozen `microsoft/rad-dino` (ViT-B, self-supervised on 838k CXRs) + a small trained head** over full fine-tuning — counterintuitive until you reason it through: with only ~16k *site-biased* TB images, unfreezing 86M parameters would let the model memorize *which hospital* a film came from and collapse out-of-distribution. The frozen head physically can't overfit to site noise. It's also what makes the whole thing tractable on a MacBook M4: extract features once, train the head in seconds.

The user then pushed for **best quality, not just binary** — the right instinct. I expanded the design to a **multi-task, multi-output** model on a fused feature (Rad-DINO ⊕ a supervised TorchXRayVision expert, the latter giving multi-pathology signal for free, no extra data): TB activity subtype (active vs latent), binary TB probability, pathology multi-label, and box-supervised lesion localization — with the calibration/conformal layer, test-time augmentation, and feature-space OOD detection on top. Multi-task supervision should *raise* the primary TB AUC by forcing pathology-grounded rather than shortcut features.

The discipline that matters most here is the **anti-shortcut protocol**: cross-source hash dedup, lung-field segmentation/cropping (strip the markers and borders models cheat on), patient-level + leave-one-dataset-out validation as the *honest* score, a site-leak canary, and Grad-CAM verification that the model looks at lungs. Literature shows a CXR model can read age/sex/scanner at AUROC≥0.90 from one image; the entire protocol exists to stop the model from learning that instead of TB.

**Status:** datasets downloading (Qatar/Montgomery/Shenzhen/TBX11K + lung masks), Python env ready (torch + MPS, both frozen backbones), pipeline scripts written. Next: fused feature extraction → multi-task leave-one-dataset-out training → anti-shortcut audits → multi-output ONNX export → wire into the app's `ensemble.tb` slot → validate.

## Milestone 8 — Verification pass: two reviewers, and the fail-open that mattered (2026-05-24)

Before building further I ran a deliberate QA pass — a code-review agent across the whole codebase, then an **independent GPT-5.5 review** as a second opinion. The agent verified the core math and caught two real bugs: a calibrated-vs-raw probability mismatch in the VLM safety threshold, and a degenerate evaluation split. Then GPT caught the one the agent missed, and it was the one that mattered most: **if every perception model failed at once, the legacy fused score defaulted to 0, and the system would return a confident "NO TB"** — a fail-open on total perception failure, the worst possible direction for a screen.

I fixed it so zero perception signal can never clear a patient (forced abstain), hardened the adjudicator to coerce malformed verdicts/confidence to a safe abstain, added terminal stage statuses so the UI can't hang on an errored stage, and tightened several training-script robustness points. Everything re-verified: strict build clean, 11/11 tests green, Python compiles and imports.

**Lesson:** two independent reviewers with different blind spots beat one — and in safety-critical code, the bug that matters is usually the **fail-open you didn't think to test for**. I didn't have a test for "all models down at once"; GPT reasoned about it from the code. That's now a guarded path.

## Milestone 9 — An expert panel red-teams the science (2026-05-24)

After the code was verified, I convened a six-lens validation panel — ML methodologist, thoracic radiologist, TB epidemiologist/patient advocate, a two-way steelman, a literature-evidence agent, and a red-team auditor — to validate the *science*, not the code. The convergence across independent perspectives was the valuable part, and it was humbling in the right way:

- The literature lens **confirmed the architecture** (frozen Rad-DINO probe, leave-one-dataset-out, threshold calibration, lung-segmentation) is evidence-supported, and that my expected ~0.80–0.88 LODO AUC sits in the *same band as commercial CAD4TB/qXR/Lunit on external data*. The engineering bet is sound.
- But three or more agents independently flagged the same overclaims: the conformal "≥90% sensitivity guarantee" **doesn't survive deployment shift** (it's an in-distribution, fit-on-same-data, small-N guarantee); "active vs latent TB" from one film is **radiologically incoherent** (latent TB is defined by a *normal* radiograph); my labels are **radiographic, not bacteriological**, so I can't map them to the WHO bar; and a 30-image eval can't support a 90% claim (you need ~150+ positives). The public-health lens added the number I'd omitted: at 1% prevalence, 90/70 means ~3% PPV and ~33 confirmatory tests per case found.

I folded every correction into the plan — softened the guarantee language to "calibrated operating point, re-fit per site, reported with CIs," relabeled the incoherent class, mandated reporting against radiographic labels, switched lung-cropping to a dilated soft-mask that preserves hila/pleura/apices, hardened the dedup + leave-one-dataset-out against cross-source patient leakage, and rewrote the ship-gate around the one experiment that actually predicts deployment (realized sensitivity at a *frozen* threshold on a held-out site).

**Lesson:** my code was correct; my *claims* were ahead of my evidence. A panel of adversarial domain experts caught the gap between "the math runs" and "the math means what I said it means." For anything that touches a clinical decision, that gap is the whole game — and the fix wasn't more model, it was more honesty.

## Milestone 10 — First real numbers from the trained model (2026-05-24)

After all the planning and review, the baseline landed — the moment the project stopped being scaffolding and started being a model. On a balanced 2,177-image subsample of three open sources (Qatar/Montgomery/Shenzhen), leave-one-dataset-out:

- **Fusion-only** (Rad-DINO CLS + TorchXRayVision): mean LODO AUC **0.858**.
- **+ patch-attention head**: mean LODO AUC **0.897** — a measured **+0.040** from the attention lever, matching the literature's prediction. (That I'd shipped CLS-only first was the mistake; the ablation proved the fix.)

That 0.90 sits in the **same external-AUC band as commercial CAD4TB/qXR/Lunit** — from a frozen-backbone probe on an afternoon's open data, versus the 42% the VLM-only managed.

But the honest reading is in the per-fold detail. On Qatar and Montgomery the frozen (train-derived) threshold *transferred* — sensitivity 78–82% at >90% specificity. On **Shenzhen it did not**: AUC 0.85 but only **31% sensitivity** at that frozen threshold, because Shenzhen's score distribution is shifted and the cutoff was too high. That's the textbook "fixed thresholds don't transfer across sites" result the commercial literature warns about — and the exact reason the method reports *frozen-threshold cross-source* sensitivity instead of a flattering in-fold number: it makes the transfer gap a measurement, not a worry.

Caveats kept loud: Qatar is a re-mix of the NLM sets, so residual overlap likely **inflates** 0.90 above a true-external figure (pHash dedup removes copies, not deeper spectral overlap); labels are radiographic, not bacteriological; it's a subsample. So 0.90 is an encouraging upper-ish bound, not the deployment number — the full run (+TBX11K, leakage-grouped LODO, confirmed-label tier) gives the truer one.

**Lesson:** the first real measurement did three things at once — confirmed the architecture works (commercial-band AUC), *proved* the attention lever (+0.04), and surfaced the operating-point-transfer problem as a number rather than a hypothesis. Recorded to `docs/baselines/` as the comparison point for everything that follows.

## Milestone 11 — The Shenzhen "collapse" was a threshold, not the model (2026-05-24)

The baseline's Shenzhen fold looked alarming: 31% sensitivity. The instinct is to throw data at it. I checked first whether it was a *discrimination* problem or a *calibration* one — because the AUC (0.82–0.85) didn't match a 31% sensitivity. A model that ranks TB vs normal at 0.82 AUC but catches only a third of cases isn't weak; it's sitting at the wrong operating point.

So I changed the honest scoreboard: every LODO fold now reports **two** sensitivities — **cold-start** (the frozen train-derived threshold, no local adaptation) and **+ local recalibration** (threshold fit on a small disjoint slice of the held-out site, evaluated on the rest, simulating "deploy with a little local labeled data"). The result settled it:

| Held-out site | Cold-start sens | + local recalibration | Spec cost |
|---|---|---|---|
| Shenzhen | 33.9% | **89.5%** [0.85–0.93] | 99% → 41% |
| Montgomery | 67.2% | 97.6% (n=41, wide CI) | 96% → 20% |
| Qatar | 78.8% | 92.4% | 95% → 59% |

Shenzhen wasn't broken — its threshold hadn't travelled. With a local calibration slice it hits 90% sensitivity, exactly what AUC 0.82 should permit. But the dual report also makes the *price* honest: reaching 90% sensitivity on Shenzhen drops specificity to 41% — a flood of false positives — because at this AUC you cannot have both. That specificity collapse is the real argument for more and better data: a higher AUC is what buys you 90% sensitivity *cheaply* (at acceptable specificity), and no amount of threshold tuning substitutes for it.

I also caught that head training was unseeded (~0.02 AUC run-to-run; the attention-mean wobbled 0.897 → 0.886 between runs) and pinned the seed, while noting the truthful figure is a band, not a point.

**Lesson:** when a safety-critical metric looks catastrophic, separate discrimination (AUC) from operating point (sensitivity at a threshold) *before* reaching for more data. Reporting the pair — cold-start floor and recalibrated-achievable, with the specificity each costs — tells the whole truth where a single number would have lied in one direction or the other.

## Milestone 12 — A reliability audit, and the day the honest numbers got harder (2026-05-24)

One line from the user — *"are the datasets normalized, will the different fields mess it up?"* — became the most important correction of the project. I ran a **site-leak canary** (can a trivial classifier name the *source dataset* from the frozen features?). It scored **balanced accuracy 1.000** against a 0.333 chance. The features were perfectly site-separable, which meant every LODO number to date was suspect: the head could be reading scanner/resolution, not TB.

So I put the whole pipeline through a **four-agent literature audit** (data harmonization, evaluation methodology, ML/calibration math, foundation-model input contracts) and then had **gpt-5.5 steelman** my proposed fixes. The convergent verdict was humbling and clarifying.

**The plumbing was right.** No register-token bug in Rad-DINO, the patch-grid slice and the mean/std were correct, the attention-MIL math and the conformal index were sound. The foundation held.

**But two of my own planned fixes were wrong turns,** and the steelman caught them. I wanted to **Gaussian-blur** images to erase the resolution shortcut — gpt-5.5 pointed out I'd be suppressing the exact high-frequency signal TB sensitivity depends on (miliary nodules, faint apical infiltrates), trading away the safety-critical metric to make a diagnostic number prettier. And **chasing the site-canary to chance** was optimizing the wrong thing: foundation features always encode acquisition; the goal is external validity and threshold transfer, not site-invisible features. I cut both. The discipline that mattered most was deleting my own clever idea.

**What I kept and shipped:** dropped CLAHE for a monotonic normalization closer to Rad-DINO's pretraining; antialiased resolution standardization (no blur); detect-and-*log* inversion (no silent pixel-flipping — medical-grade needs the DICOM tag); true TorchXRayVision logits; a cross-source **provenance match-graph** in dedup with a leak-guard so re-mixed wrappers can't span the train/test split, and cross-label clusters raised as a data-quality alarm; dropped label smoothing (it was distorting the very probability we fuse); **temperature calibration with ECE reported**; **bootstrap AUROC CIs**; and **PPV/NPV with confirmatory-tests-per-case at 1–2% prevalence**.

That last number matters most and flatters least: at a 0.93-sensitivity operating point, **1% prevalence gives ~2% PPV and ~48 confirmatory tests per flagged case**. A model can post 0.90 AUROC and still send 48 people for confirmation per true case found. That's the honest cost of screening at low prevalence, and it's in the report now.

**The hardest correction was semantic, not numerical:** my labels are *radiographic*, not microbiological. The system detects a *radiographic pattern associated with TB labels* — not confirmed *active* TB. I reframed the endpoint everywhere to say exactly that.

**And the "medical-grade" question.** The user asked for a medical-grade system. The honest answer, which gpt-5.5 put bluntly and I have to record: you cannot get there by patching a frontend BYOK prototype on open radiographic data. Medical-grade needs microbiological ground truth, prospective multi-site validation, a locked server-side model (not a browser calling third-party APIs that can change under you), an LLM kept *out* of the decision path, and a QMS/regulatory program (ISO 13485, IEC 62304, ISO 14971, an FDA/CE pathway). What this project can honestly be: *built with medical-grade engineering discipline, clearly labeled not a device, with a documented path to clearance.* I'd rather ship that sentence than overclaim.

Measured after the fixes (2,177-image subset, backbones unchanged): fusion-only **0.871 → +attention 0.905** LODO AUROC — the attention lever survived every correction. The full 4-source re-run on the harmonized pipeline is the next number.

**Lesson:** a one-line user question was worth more than any metric. The site-leak canary is the cheapest honest instrument I added — and reliability came from being willing to delete my own clever fix when the steelman showed it traded sensitivity for a prettier diagnostic.

## Milestone 13 — The full validated run, and the attention lever deflates (2026-05-24)

The audit-fixed pipeline ran end to end on all four sources — **13,092 deduplicated images**, harmonized preprocessing, the dup-cluster leak-guard on, bootstrap CIs, prevalence-aware reporting. The honest 4-source numbers:

- **Fusion-only mean LODO AUROC 0.916; fusion+attention 0.924.** Strong, in the commercial external band — but read it through the leak flags, not the mean.
- **The attention-over-patches lever deflated to +0.008.** It looked like the headline win on the 2,177-image subset (+0.034), and the commercial literature credited patch-attention with +6 AUROC. At full scale it's inside the bootstrap CIs — marginal. More data made the CLS+TorchXRayVision fusion strong enough on its own that attention added almost nothing (it still helped TBX11K +0.024 but was neutral-to-negative on Montgomery). I'm recording the deflation, not the earlier hope: at this scale, attention pooling is a rounding error here, not the lever I'd billed since M9b.
- **Shenzhen's cold-start "collapse" from M11 is gone** — 34% sensitivity at a frozen threshold → **79%** on the full run. Harmonized intensity + dropping label smoothing + temperature scaling + more training data fixed the threshold transfer that a local recalibration had previously been needed to rescue.
- **Site-leak fell 1.000 → 0.945** (4 sources, chance 0.25): the features are still highly site-separable, exactly as the steelman warned — so I trust the cleaner-external folds (Montgomery AUROC **0.88**, Shenzhen **0.94**) and treat the re-mix folds (Qatar/TBX11K, 0.94) as leakage-inflated.
- **The deployment number, on the cleaner externals** (sens 0.94, spec 0.50): at 1% prevalence, **1.9% PPV and ~54 confirmatory tests per flagged case**; cross-site calibration still imperfect (Montgomery ECE 0.21). A radiographic-pattern detector, not an active-TB diagnosis.

**Lesson:** the subset lied about the attention lever; the full run + bootstrap CIs caught it. The cheapest insurance against a flattering result is more held-out data and a confidence interval — plus the willingness to write down "the thing I thought mattered, doesn't."

## Milestone 14 — A review swarm, and the correction I owed (2026-05-24)

Before building the T2 "sharpening" heads, I put the work through a deliberate gauntlet: gpt-5.5 + 4 code/methodology reviewers (debugger, ML engineer, code-quality, architecture) + 3 data auditors (form, original→vectorized correctness, training-fitness). It caught two silent bugs and one of *my* overclaims.

**Two coordinate bugs the unit tests couldn't see.** The Rad-DINO processor does shortest-edge-resize **+ center-crop** to 518², so for non-square crops the patch tokens covered only the centre square while the box/zone labels were built over the full cropped rectangle — a systematic misregistration that would have corrupted exactly the apical/costophrenic supervision, and ran clean through 9 square-crop unit tests. Fixed by letterboxing image+masks+boxes to a common square. The second was subtler: the token-order assert was a *tautology* (it only proved numpy reshapes row-major, not that Rad-DINO *emits* row-major). I added a real empirical test — feed a white-top/black-bottom image through the live model — and it **confirmed Rad-DINO is row-major** (token norms separate ~6.4× more along rows than columns). The box/zone geometry is now verified, not assumed.

**The correction I owed.** In M13 I read the label-randomization result (0.505 ≈ chance) as proof the model uses "genuine TB signal, not a site shortcut." The methodology reviewer showed that doesn't follow, and the data auditor proved why with numbers: **site is 98% recoverable from the features and TB prevalence swings 6.3× across sources** (tbx11k 8% → shenzhen 51%). Because site and prevalence are confounded, a model exploiting "this looks like Shenzhen → 51% prior" would *also* collapse under label permutation — so 0.505 only rules out label *memorization*, not site-keyed shortcutting. The honest position: the randomization probe is necessary-not-sufficient, and the **lesion-localization harness + provenance-independent external validity are the real discriminators**. I'd rather record the walk-back than keep the flattering reading.

**What the data audit also settled — some of it reassuring.** The vectorized data is well-formed (no NaN/inf; the 18 TorchXRayVision values are真 logits, not sigmoid'd) and the original→features combination is faithful (0 label/alignment mismatches across 13,092 rows). And LODO is a *genuine* external test: the TB **positives are disjoint across sources** (Qatar-positive vs NLM max cosine 0.78; cross-fold leakage ≤0.12% and entirely negatives) — the re-mix-leakage fear is smaller than I'd assumed at the positive level. The standing risk is the site-confound, not positive leakage.

**The plan got re-sequenced.** I had labelled the box-supervised head "the #1 lever." The ML review (and my own §7) say it isn't: all box supervision is 100% TBX11K (a leakage-prone fold), so it buys interpretability and a faithful evidence map, not external recall. The real external-sensitivity levers, re-ranked: **the native 37×37 patch grid + letterbox** (our 8×8 pool averages a TB focus into a 65px cell, discarding the subtle/miliary/subclinical signal that caps sensitivity), then the **lower-zone-floored zonal soft-OR**, then **active-vs-healed** for usable specificity. And a process guardrail: with T2's many loss-weight knobs, tuning any of them against the LODO test folds would optimistically bias the headline — so weights are prior-fixed / nested-CV'd, never tuned on the test fold.

**Safety correction to the verdict.** A confident **NO** on a subclinically-positive, normal-looking film is the one place the verdict layer can falsely reassure with nothing in the image to flag (~40–50% of prevalent TB is subclinical). No guardrail can catch it, so every NO now carries a hard-wired *"cannot rule out subclinical/early TB — test symptomatic patients regardless."*

**Lesson:** a review swarm earns its cost when it deletes your own clever idea (the blur, twice), catches a bug your tests structurally can't (square-only crops), and makes you walk back a conclusion you'd already written down (0.505 ≠ "no shortcut"). The honesty infrastructure works only if you let it correct *you*, not just the code.

## Milestone 15 — The T2 sharpening heads, and what "modest AUROC" actually bought (2026-05-24)

I built the three heads the blueprint called for and put them through the same LODO harness, prior-fixed every knob (g_min=0.25, box-pool sharpness r0=5 capped at 8, λ_box=0.2, λ_distill=0.2) so nothing was tuned against the test folds, and ran each lever as its own ablation. The headline I'd internalised from M14 was right and worth restating before the numbers: **these heads were never going to move AUROC much** — the open-data radiographic ceiling is ~0.92–0.95 and the modality itself is the limit. So I set the success criterion *before* the run: worst-fold external sensitivity and lesion localization, not the mean AUC. I also wrote a drift tripwire — if FULL AUROC jumped >+0.05, I'd treat it as a leak, not a win.

**What the levers did.** Same-run re-baseline: fusion-only 0.914, +attention 0.934 (a touch below the logged 0.922/0.938 — MPS run-to-run noise, same band). Then: pathology-distillation alone **0.912** (AUROC-neutral, slightly down — exactly as expected; distillation anchors the representation in named findings to fight site-shortcutting, it isn't an AUROC lever). The **zonal soft-OR pushed worst-fold Shenzhen recalibrated sensitivity 0.939 → 0.953**, and the **FULL head reached 0.958 [0.92-0.98]** — the single number I most cared about, because the soft-OR (logsumexp across the 7 floored-gate zones) is exactly the mechanism that lets a lone lower-zone lesion fire the image logit, which is the documented South-Asian diabetic-TB miss. Mean FULL AUROC was **0.925, −0.009 vs +attention** — no jump, tripwire silent, drift flag clear.

**The box head, and the honesty I owed it.** All 660 boxes are TBX11K — a leakage-prone re-mix fold — so a naive "box head improves AUROC" claim would be exactly the kind of thing the M14 review caught me on. So I reported the box lever's mean AUROC *over the folds whose held-out site has zero boxes* (Montgomery/Shenzhen/Qatar): **0.924, essentially identical to the overall 0.925** — the box supervision is not leaking recall in through TBX11K. What the box head actually buys is the evidence map, and I validated it the honest way: train on the non-TBX11K sources (which have **no boxes at all**, so the cell loss never fires in that fold) and score the map against the held-out TBX11K boxes. **Pointing-game hit-rate 0.865 [0.84-0.89], mIoU(top-k) 0.604 [0.58-0.63], against a random-attention floor of 0.105/0.059** — a +0.76 / +0.55 lift. That the map localises lesions it was never given a single box for, on a held-out site, is the strongest anti-shortcut evidence the project has produced — it's pointing at the pathology, not the scanner. Caveat I keep attached to it: the grid is 8×8 (pooled from Rad-DINO's 37×37), so mIoU is a *relative-to-random tracking number*, not an absolute localization claim — the native 37×37 grid is still the real fix.

**Lesson:** when you decide the success metric before the run and it isn't the flattering one, the result reads honestly no matter which way it lands. AUROC barely moved (by design), and the win showed up where the blueprint said it would — worst-fold sensitivity (+0.019) and faithful localization (+0.76 pointing-game over random) — which is the kind of improvement a screen is actually judged on. The levers that helped were zonal soft-OR (sensitivity) and box-evidence (localization + operating-point specificity); distillation paid its way as a grounding anchor, not an AUROC bump.

## Milestone 16 — The old-scar head, and a number that proved a gate was the wrong tool (2026-05-24)

I set out to build the active-vs-healed head the blueprint had ranked as the specificity lever for South Asia, where the dominant false-positive is an **old, healed TB scar** — calcification, fibrosis, volume loss — misread as active disease. I had a 139-image inactive-sequelae probe (the TBX11K rows the dataset mislabels as "latent"; I corrected that misnomer up front — true immunologic latent TB is radiographically *silent*, so I never let the system emit "latent TB" from a film; the score is an inactive/sequelae *pattern* score). First correction to record: the blueprint expected the probe to dedup to 169 unique images. The current index has **139** after dedup, so 139 is the denominator I report — not the number the plan anticipated.

**The head, and why the patch tokens stay in.** A small auxiliary head (gated-attention over Rad-DINO patches + CLS + the 18 TorchXRayVision logits), positive = the 139 sequelae, negative = active TB. I deliberately kept the Rad-DINO patch tokens in even though the 18 named-finding logits are cheaper: TorchXRayVision has **no Calcification logit**, and calcification is the single most specific inactive sign — a head on the 18 logits alone would be blind to exactly the feature that matters. The probe was never in the main `features.npz`, so it's automatically held out of the TB head's training and eval.

**The smoking gun.** The honest deliverable is SeqFPR — how often the main TB head flags an old scar as screen-positive at its operating threshold. **SeqFPR = 117/139 = 0.842 [95% Clopper-Pearson 0.770–0.898].** The head over-calls 84% of healed scars as active, and it isn't hedging: **72% of the sequelae score p≥0.95**, median 0.998. That single number is the clearest statement of the South-Asian specificity problem the project has produced.

**The active-vs-sequelae AUROC, reported the honest way.** The probe and the active-TB boxes both come from TBX11K, so a high active-vs-healed AUROC could be a TBX11K-internal split rather than a transferable signal — I flagged >0.90 as suspect-leakage before running. Using the head's `s_inactive` with **TBX11K-only negatives (same provenance — the hard, honest test): AUROC 0.722 [0.67–0.77]**; with all-source negatives (optimistic, easier): 0.860 [0.83–0.89]. Both sit at or below the ~0.82–0.88 ceiling for active-vs-healed on a single frontal film, neither trips the leakage flag — but the shared-provenance confound stays attached to the claim: this may not transfer off-site. (The main TB prob's AUROC for the sequelae label was 0.34 / 0.28 — inverted, as it should be, since the head scores scar *high*.)

**The result that changed the design.** I built the escalate-not-clear verdict gate the blueprint asked for: high `s_inactive` + a weak active signal forces `activity: indeterminate` → UNDETERMINED, and the inactive head can RAISE old-scar suspicion but never CLEAR a flagged film (reactivation hides in scar). The gate's truth table is exactly right. But when I measured the *net SeqFPR reduction* from gating, it was ~zero — and the reason is the load-bearing finding: the main head is **confidently** wrong on scar (72% at p≥0.95), and an escalate-not-clear gate by construction must not override a confident active signal. A downstream gate therefore **cannot** fix SeqFPR. The real fix is to feed `s_inactive` into the head as a specificity *feature* / recalibration input, or to hard-negative-mine old scar into the active head's training — not to bolt a gate on the end. The gate's genuine value is the *inverse* direction: on a weak active signal, raising old-scar suspicion so a subtle reactivation in a scarred lung gets reviewed instead of cleared.

**Lesson.** I asked for a gate and the measurement told me a gate solves the wrong half of the problem. The over-call isn't a borderline case the gate can catch — it's a high-confidence error baked into the active head, which only training-time signal (a feature, a recalibration, hard negatives) can move. Reporting the ~zero net reduction honestly, rather than quietly shipping a gate that looks like it addresses SeqFPR, is the point of the whole exercise. The head isn't ONNX-exported into the app yet; that wiring is the follow-on, and it should wire `s_inactive` as a feature, not just a gate.

## Milestone 17 — Multimodal fusion, and proving the machinery without the data to claim a result (2026-05-24)

The brief was a multimodal fusion prototype: fuse the image score with optional clinical inputs (age, sex, cough duration, fever, weight loss, night sweats, **CRP**, HIV status, diabetes, prior TB, TB contact), with a hard requirement — **the clinical inputs are optional**. If a patient arrives with no clinical data, the system must *still score from the image alone*, just *less confidently*; when the clinical fields are present, they should sharpen the decision and raise confidence. The honest constraint behind the whole exercise: we have **no paired image+clinical+outcome data**. That pairing is gated (R2D2's tabular cohort needs images by outreach; TB Portals and the India survey microdata need a DUA — `DATA_SOURCES.md` §6-8). So this milestone is explicitly a **machinery proof**, not a performance claim, and the design had to make that boundary impossible to blur.

**The design — late fusion in log-odds.** `fused_logit = image_logit + g · clinical_logit`. The image logit is always computed; a small clinical-MLP produces a clinical logit from a `ClinicalVector` (each field carries a normalised value *and* a PRESENT/ABSENT mask bit, fed as `[value·mask ; mask]` so the MLP can see which fields are missing). The gain `g = softplus(·)` is non-negative by construction, so clinical can only *sharpen* the image evidence, never invert it — the deterministic-safety-net ethos, in the architecture. When the whole clinical vector is absent, the input is all zeros and the fused logit reduces exactly to the image logit. The mechanism that makes image-only a *first-class, calibrated* pathway rather than a degenerate fallback is **modality dropout**: I drop the entire clinical vector for 50% of training samples, so the model is forced to learn both pathways and the image-only path stays calibrated. Confidence is operationalised with a per-regime conformal band `[tauLow, tauHigh]` (the same band the app's `calibration.ts` fits): a case is UNDETERMINED when its score lands inside the band, and the band is fit *separately* for clinical-present vs clinical-absent — so the rule "less confident without clinical" is a measured band width, not a slogan.

**The bug that would have faked the whole thing.** My first run reported image AUROC **1.000**, specificity 1.000 at 90% sensitivity, and a conformal band of width *zero* — every machinery check "passed" and every number was a lie. The cause was the M13/M15 lesson I should have remembered: I scored the trained T2 head **in-sample**, on the very images it was trained on, so it had memorised them. There was no generalisation gap for the clinical signal to act on and nothing for the band to be uncertain about. The fix was to score the image side **out-of-fold**: retrain the T2 head leave-one-dataset-out and take each image's logit from a head that never saw its source. That restored the honest **OOF AUROC 0.924** — exactly the M15 LODO baseline (0.925), confirming no leak — and gave the fusion the real ~8-point headroom it needs to demonstrate anything. Faked headroom would have made the clinical layer look magical; the real headroom is what makes the demonstration mean something.

**The clinical side, simulated loudly and calibrated to the literature.** With no real paired data, I simulate the clinical signal from label-conditioned Gaussians, with per-field weights shaped by the TB literature (CRP and prolonged cough carry the most, sex the least) and an effect size **calibrated to R2D2**: at fixed 90% sensitivity, R2D2 reports specificity rising 70.3% → 75.9% (+5.6 pts) when CRP is fused into a CAD score. I swept the one simulation knob against that target (0.75/0.85/0.95 → +0.047/+0.055/+0.061 spec gain) and fixed it at 0.85. Crucially I also honoured R2D2's *other* finding — fusion gains **shrink as the image model strengthens** (+0.07 AUC on a weak model, +0.01 on a strong one) — so a single-digit specificity gain on our already-strong 0.92 image model is the *expected, honest* outcome, not a disappointing one. Every print statement and the docstring say SIMULATED; the brief's whole point is that I must not let a simulated coefficient masquerade as a measured multimodal accuracy.

**The three machinery results (image REAL, clinical SIMULATED).** (a) **Full-modality ≥ image-only**: specificity at the 90% sensitivity floor rose **0.834 → 0.875 (+0.040**, AUROC 0.926 → 0.952) — single-digit, R2D2-consistent (it wobbles ~±0.015 run-to-run on MPS, so the *band* is the honest read, not a decimal). (b) **Graceful degradation**: with clinical absent the system still scores, AUROC **0.926** on 3,143 held-out cases — the fused logit cleanly becomes the image logit. (c) **Less confident without clinical**: the conformal band is **0.230 wide in image-only mode vs 0.141 with clinical**, and the UNDETERMINED rate rises **0.043 → 0.092** when clinical is missing. The system never refuses to score; it abstains more, on purpose, when it knows less.

**Lesson.** The honest move here wasn't a number — it was building the apparatus so that the absence of real data is structural, not hidden. The image side is real and out-of-fold (I caught myself faking its headroom and fixed it); the clinical side is simulated and labelled as such in every output; the effect size is pinned to a citation, not chosen to flatter; and the result I report is "the machinery works and degrades correctly," not "multimodal is X% accurate." The next step is data acquisition (R2D2 images / a TB Portals DUA), not more modelling — and the frontend clinical form is a further follow-on, gated on an ONNX export of this head, because the app today calls API providers, not the trained head.

## Milestone 18 — The NIH stress test, and the failure mode being smaller than I feared (2026-05-24)

After the deployment-gates work made the catastrophic scar-subgroup calibration visible — `tbx11k_scar` Brier **0.806**, ECE **0.846**, FPR **0.842** at the deployed 0.95-sensitivity threshold — I had an honest question I could not answer from a single 139-image probe: is the decision layer *broadly* broken on TB radiographic mimics (fibrosis, nodule, consolidation, mass, pleural thickening, infiltration, atelectasis), or is the calibration disaster *specifically* about confidently over-calling old scars from the same provenance as the training positives? The answer changes the roadmap. If broad, the decision layer needs a rebuild; if scar-specific, the sequelae head is the targeted fix and we keep moving. I needed a sample-rich, labelled-per-finding, *off-provenance* mimic set to settle it. NIH ChestX-ray14 — 10,000 images, 15 per-finding labels, a different institution and scanner mix than any of our training sources — was the right tool.

**The design — measure through the IDENTICAL preprocessing.** The whole point of a stress test is that the *only* thing that changes is the input distribution. So `training/extract_features_nih.py` imports `_read_gray`, `_harmonize`, `seg_crop`'s lung-mask + letterbox-to-square, and `zone_matrix_from_masks` directly from `training/extract_features.py`. Every constant — WORKING_RES=1024, PCLIP=(1,99), PATCH_GRID=8, BATCH=16, WORKERS=6 — is reused. No new variants. The trickiest call was `zones`: NIH has no boxes (so `grid_label`/`has_box` legitimately drop), but `zones` is a *structural* input to `ZonalSoftOR`'s forward pass — not a label. Omitting it would force me to feed all-zero zones at inference, which silently collapses the zonal lever and changes the model I am measuring. I computed zones from the segmenter (no NIH labels needed) and noted the decision in the code header and the commit, because under the "deployed-style" rule you measure the actual deployed forward pass or you measure nothing useful. Extraction ran 30 minutes on the M4 (0.28 s/img), produced a 942 MB npz with the schema `features.npz` uses minus the box-supervision tensors plus the 15 per-finding label columns.

**The threshold and temperature are the deployed ones — re-derived, not read.** `training/nih_stress_run.py` loads the cached `tb_head_t2.pt` (fusion + zonal + box, the prior-fixed config from `train_tb.py`'s final save), then re-fits T and the 0.95-sensitivity threshold on the *same* SEED=0 0.15 stratified validation split `train_tb.py` uses for its final save. That gave T = **1.5915**, thr = **0.6105**, |ΔT|=|Δthr|=0.0000 vs `tb_threshold_t2.json` (the EXPERIMENT_LOG row from the scar stress test). Letting the data redrive these and confirming they land on the saved literals keeps the audit trail consistent without trusting a cached number.

**The expected answer, before I ran it.** I wrote my prior in the experiment log row: TB-mimic subgroups will show high FPR and bad Brier/ECE, *similar to the scar probe*; No_Finding will be lower; the gap is the headline. Direction expected; magnitude unknown — Brier ≈ 0.5–0.8 on mimics would be a broad decision-layer failure; Brier ≪ 0.806 on NIH mimics but with a clear mimic-vs-easy gap would be scar-specific.

**The result — direction confirmed, magnitude not.** The two worst NIH findings are exactly the scar-shaped ones, in the right order: **Pleural_Thickening** n=273, FPR **0.103 [0.069–0.145]**, Brier **0.088**, ECE **0.124**; **Fibrosis** n=147, FPR **0.102 [0.058–0.163]**, Brier **0.102**, ECE **0.149**. Then Mass (FPR 0.067), Nodule (0.063), Consolidation (0.049), Infiltration (0.040), Atelectasis (0.018). No_Finding sits at FPR **0.016**, Brier **0.016**, ECE **0.030**. Cardiomegaly — anatomical, not parenchymal — sits at 0.011. Overall NIH FPR across all 10k images is **2.62%** — comparable to the deployed head's in-distribution specificity. The within-NIH mimic-vs-easy gap is real: TB-mimic mean FPR 0.063 / Brier 0.059 / ECE 0.097 vs No_Finding 0.016 / 0.016 / 0.030 — about a 4× gap, and ~6–7× for the two worst classes. But the *magnitude* is nowhere near scar: **scar Brier 0.806 is ~8× the worst NIH-mimic Brier (Fibrosis 0.102); scar ECE 0.846 is ~6× Fibrosis ECE 0.149; scar FPR 0.842 is ~8× Pleural_Thickening FPR 0.103.** The shape of the failure carried — radiographically scar-shaped findings get over-called — but the disaster did not.

**Why the scar probe is uniquely bad — the read I actually trust.** Two readings line up. (1) The TBX11K healed-scar probe shares provenance with the *training positives*: an active-TB image and a healed-scar image from the same hospital, scanner, and population. A model that learned "active TB at this site looks like X" will be confidently wrong on a healed scar from the same site that also looks like X. NIH is a different institution and scanner mix — the same shortcut does not apply, so confidently-wrong is replaced by *mildly* over-confident on the same radiographic shapes. The frac≥0.95 numbers make this stark: on scar **0.719** of predictions are p_active ≥ 0.95; on Fibrosis it is **0.048**. The head is not catastrophically broken — it has a *site-anchored* confidently-wrong failure that does not generalize. (2) The deployed head transferred to NIH with a 2.62% naive FPR — comparable to its in-distribution specificity — which is the opposite of an off-distribution collapse.

**What this changes in the roadmap.** The decision is to keep the sequelae head (`InactiveSequelaeHead` + `s_inactive` as a recalibration *feature*, plus the escalate-not-clear gate's inverse-safety direction) as the *targeted* scar-mimic fix, and to **not** rebuild the decision layer for broader mimic discrimination. The data does not support that the broader decision layer is broken. The honest framing for any future writing is: the head is *mildly* over-confident on radiographically scar-shaped findings (Fibrosis FPR ~10%, ECE ~0.15) — a real issue worth disclosing — and *catastrophically* over-confident on same-provenance healed scars (FPR 84%) — the targeted fix. Two issues, two magnitudes, two fixes.

**Lesson.** When a single 139-image probe gives you a Brier of 0.806, the temptation is to call the decision layer broken and reach for a big rebuild. The honest move is to test that hypothesis on a sample-rich, off-provenance set *through the same preprocessing*, and let the magnitude tell you whether the fix needs to be targeted or systemic. NIH said targeted. The corollary — that a small same-provenance probe can wildly over-state a decision-layer failure — is itself something to remember; the scar Brier wasn't lying about the failure mode, but it was lying about the size of the surface the failure covers.

## Milestone 19 — Shipping the sequelae fix as far as I could ship it (2026-05-24)

The NIH stress test in M18 told me the catastrophic calibration is *scar-specific* — TBX11K healed scars FPR 0.842, Brier 0.806, ECE 0.846, while NIH Fibrosis sits at FPR 0.102 / Brier 0.102. That settled the roadmap: the targeted fix is the sequelae head's `s_inactive` as an escalate-not-clear feature, not a broader decision-layer rebuild. So I set out to ship that fix as far as it could *honestly* ship without first solving the heavier Phase-B question (running Rad-DINO + TorchXRayVision on real images in the browser), and to be careful about which parts I was claiming to have shipped.

**The scope decision, made up front.** Three pieces are shippable today: (1) an ONNX export of both deployed heads + a calibration sidecar, (2) a parity gate that proves the ONNX behaves identically to the trained PyTorch, and (3) an orchestrator interface that *accepts* an optional `s_inactive` and applies the escalate-not-clear rule when it's present. One piece is not: actually *producing* `s_inactive` at inference time in the browser. The reason is concrete — both heads consume Rad-DINO patch + CLS tokens and TXRV features, which means the browser would need to run two heavy backbones (Rad-DINO ~600 MB, TXRV DenseNet ~30 MB) with different preprocessing pipelines, or call an HF feature-extraction endpoint for each, or vendor a distilled small backbone. Each option is a real piece of work with its own design tradeoff; the M19 brief was scoped to the export + interface, so I made the gap a first-class fact in the code, the docs, and the safety-net contract rather than papering over it.

**The export, and the part I almost shipped wrong.** I wrapped `TBHeadT2` so the `zones` tensor — set as a side-channel attribute in training (`self._zones`) — becomes an explicit forward arg the exporter can trace. I baked the sequelae head's temperature (T=1.1313) into its ONNX graph so it emits a probability directly, but I deliberately left the main head's temperature *out* of its graph because the orchestrator owns the calibration policy (T and `thr_at_95sens` arrive via `head_calibration.json` and can be re-fit per site without re-exporting weights). My first try emitted opset 17 ONNX that onnxruntime-web could not load — the downgrade pass produced a `noop_with_empty_axes` attribute the older op-set verifier rejects — so I bumped to opset 18 (still web-compatible). File sizes: main head 1.89 MB (135 KB graph + 1.76 MB external-data), sequelae 1.25 MB (46 KB graph + 1.20 MB external-data). The two-file format is a torch-dynamo convention; both files must ship together.

**The parity number, and what it would have meant to assert on the wrong thing.** I wrote `training/test_onnx_parity.py` (mirroring `test_stress_metrics.py`'s no-pytest convention) to run a deterministic 50-row slice through both PyTorch and onnxruntime-CPU, with strict tolerance. The honest answer: **main head probability max-abs-diff 1.18 × 10⁻⁵** (logit max-abs-diff 4.6 × 10⁻⁴ on logits up to ~30), **sequelae head s_inactive max-abs-diff 1.79 × 10⁻⁷**. I asserted on the *probability* not the raw logit because the consumer that matters — the orchestrator's safety net — only ever compares the probability to a threshold; the 1e-4 logit-level drift is float32 accumulation across the 64-token gated-attention pool + LSE-LBA box pool, and it disappears under the calibrated sigmoid. Asserting on logits would have failed at 4.6e-4 while behaviorally nothing changed; asserting on probabilities is the contract the orchestrator actually needs. (The diff is in the test's print output for the audit trail.)

**Deriving the escalation threshold from the head's own scar probe.** The new safety-net rule needed a number — at what `s_inactive` should a borderline `tb_prob` (in [0.35, thr_at_95sens=0.6105)) escalate from NO_TB to ABSTAIN? I derived it from the 139-image scar probe under the deployed temperature (T=1.1313): `s_inactive` on confirmed scars ranges 0.032–0.973, median 0.774; the 0.7-recall threshold is **0.7126** (q30 of probs, actual recall 0.6978 at that point). In plain English: roughly 70% of confirmed scars score above 0.7126. I picked this rather than the `SEQ_HIGH_DEFAULT = 0.5` used elsewhere in `train_tb.py`'s `activity_verdict` because the safety-net role here is "definitely scar-shaped" — a false escalate on a clean lung costs one extra read, a false skip on a real scar costs the very FPR disaster we're trying to fix. The rule is a pure function in `src/lib/pipeline/sequelaeEscalation.ts`, unit-tested with the three required cases plus boundary cases that pin the never-clear contract: it can only push NO_TB up to ABSTAIN; it never demotes TB or ABSTAIN, never overrides a strong active signal (the latter is the M16 lesson — reactivation hides in scar).

**The browser-side feature pathway is the Phase B gap, and the wiring degrades gracefully.** The orchestrator calls `applySequelaeEscalation(...)` today with `sInactive: null`, which is a defined no-op. Once a browser-side path produces `s_inactive` — either via HF feature-extraction for Rad-DINO + a TXRV alternative, or via a vendored small-backbone distillation — that line becomes a one-liner. Until then, the interface is locked, the rule is unit-tested, and the absence of the feature is explicit rather than hidden. The decision on which Phase B path to take is deferred pending that design.

**The verdict card now tells the user about it.** Added a small, always-visible footnote line under the existing "radiographic TB screen · not a diagnosis" tagline, on every verdict — TB, NO_TB, ABSTAIN alike: *"Higher false-positive rate (~10%) expected on radiographically scar-shaped findings (healed fibrosis, pleural thickening)."* The ~10% number is the honest cross-site magnitude from NIH (Fibrosis 0.102, Pleural Thickening 0.103). I left the existing NO_TB subclinical warning untouched — that one addresses a different failure mode (a confident NO on a normal-looking subclinically-positive film), and stacking the two on NO_TB only would have hidden the scar-shape disclosure on the TB verdict, which is exactly where a referrer needs to see it before deciding how aggressively to confirm. Vitest covers the disclosure with `react-dom/server.renderToStaticMarkup`, no jsdom dep added.

**Lesson.** The temptation when shipping a "fix" is to ship the *end-to-end* user-visible change — the moment the model behaves differently on a real image. The honest version of this milestone splits that into export + parity + interface (ship now, with the parity number proved) and the runtime feature pathway (defer with a written gap). The trade is that the live system gets one user-visible change (the disclosure line) and no behavioral change yet — *and the docs say so out loud*. The alternative — a shipped feature pathway with one of the heavy-backbone options chosen on a hurry — would have buried a real design decision under a deadline. The interface is the asset; the runtime activation is a one-line change once the backbone question is decided on its own merits.

## Milestone 20 — The silent failure on production, and the BYOK contract I had been quietly breaking (2026-05-24)

I deployed M19 to Netlify, opened the live URL on a fresh browser to take a screenshot for the writeup, dropped a chest X-ray onto the canvas, and watched the agent trace get stuck. The verdict card eventually rendered "UNCERTAIN — REFER" with a near-zero confidence ring. Nothing felt wrong at the UI layer. Then I opened DevTools and saw what was actually happening:

```
router.huggingface.co/hf-inference/models/Owos/tb-classifier  400
router.huggingface.co/hf-inference/models/keremberke/yolov8m-chest-xray-classification  400
```

Two silent 400s, one per perception head. The app fell through to a low-confidence verdict and reported it as if the model had evaluated the image and was uncertain. That is exactly the failure mode my own non-negotiable ethos forbids: "Fallback and degradation are always visible to the user, never hidden." I had shipped a screening tool that lies about whether it has perceived the input.

**The first thing I did was reproduce the 400s with a valid token.** Both default models had been in the codebase since M1, and I had assumed they were still live. They are not. With `Authorization: Bearer <my hf token>`:

```
POST router.huggingface.co/hf-inference/models/Owos/tb-classifier
  → HTTP 400 {"error":"Model not supported by provider hf-inference"}
POST router.huggingface.co/hf-inference/models/keremberke/yolov8m-chest-xray-classification
  → HTTP 400 {"error":"Model not supported by provider hf-inference"}
```

These models are *browsable* on the Hugging Face Hub but no longer *deployed* on the hf-inference free serverless router. The router has been thinning its supported-model list through 2024-2025, and ours got cut. A sweep of TB-tagged classifiers (`sukhmani1303/tuberculosis-vit-model`, `runaksh/chest_xray_tuberculosis_detection`) showed none on the router. The only `image-classification` chest-xray model I could verify HTTP 200 on, with a small PNG payload, was `codewithdark/vit-chest-xray` — a CheXpert ViT with five labels (Cardiomegaly, Edema, Consolidation, Pneumonia, No Finding). Notably, none of those is "tuberculosis." Which is itself the honest result: `parseGeneralCxrTbProb` will return 0 here because none of those labels match the TB/abnormality regex. The general head provides no TB signal on its own. I documented that explicitly in `defaults.ts` rather than papering over it with a flattering regex.

**The TB-specific default I emptied.** There is currently no TB-specific classifier I could verify on the hf-inference free serverless router. So I removed `Owos/tb-classifier` as a default and left the slot blank — BYOK: the user pastes a working slug or configures a Replicate fallback. The Settings drawer hint now warns about the router retirement explicitly. This is closer to my BYOK contract than pretending a dead default still works.

**The diagnostic the user was missing.** With the dead defaults fixed, I still had a structural problem: when an HF call fails, the only signal was a stack trace inside DevTools. I added a small in-memory `providerStatus` store (`src/store/providerStatus.ts`) backed by `useSyncExternalStore`, with each provider client reporting its last call's state on success and on every failure path. I deliberately gave the hf-inference router's "Model not supported" body its own tag (`model-unsupported`) separate from `unauthorized` and `other-error`, because the user's fix is different: update the model id in Settings, not paste a new token. The Settings drawer now shows a tiny status line under each key field — `hf: last call model-unsupported · 400 model not available on this provider — update the model id in Settings (model Owos/tb-classifier)` — so the diagnostic is visible without DevTools. The pure dispatch function is pinned with eight unit tests so a future refactor cannot quietly collapse the tags back together and re-silence this failure mode.

**The card I refused to keep rendering.** When *every* perception member errors out — no key set, every key bad, every model retired — the orchestrator still produces a verdict, currently a `mostCautious`-forced abstain. The card then shows "UNCERTAIN — REFER" with a near-zero confidence ring. The model didn't read anything. Calling that "uncertain" is the same kind of lie as the silent 400s. I added a `perception_unavailable?: boolean` to `Adjudication` (extending the existing type rather than parallelling a new one), set by the orchestrator on the `noPerception` branch in both the adjudicator-success and adjudicator-failure paths. `VerdictCard` early-returns a dedicated honest empty-state card titled "Perception unavailable" that explains there is no reading at all and links to Settings via a new `onOpenSettings` prop. The misleading scar-FPR disclosure does *not* render in that state because nothing was measured. Four new vitest cases pin the contract: the card appears for `perception_unavailable=true`, the UNCERTAIN headline does *not* appear, the scar-FPR disclosure does *not* appear, and a normal abstain with the flag false still renders the regular card.

**The top-of-app banner that disappears the moment it's earned.** A fresh visitor with no keys hits the perception pipeline and sees nothing happen visibly. I added a calm `NoKeysBanner` above the existing SafetyBanner: "Add at least one API key in Settings — perception runs in your browser using your keys (BYOK). No keys, no perception." Two unit tests pin the contract (renders at `hasAny=false`, returns empty markup at `hasAny=true`), with the component split into a hook wrapper and a pure `NoKeysBannerView` so the test can render via `react-dom/server` without needing a node-env localStorage shim. I deliberately did not stack this banner on configured users — the BYOK contract is about onboarding clarity, not about nagging someone who already gave the app their keys.

**The numbers.** Vitest grew from **25 → 46 passing** across **6 files** (calibration 11, sequelaeEscalation 10, VerdictCard 8, NoKeysBanner 2, ProviderStatusBadge 7, providerStatus 8). Build clean (`npm run build` tsc + vite, no warnings except a chunk-size suggestion unrelated to this branch). Training tests untouched (6/6 stress_metrics + 3/3 onnx_parity from M19 still on disk). The one HF model I verified live as the new default: `codewithdark/vit-chest-xray` (HTTP 200, 5-label ViT, CheXpert-fine-tuned, ~5k downloads on the Hub).

**Lesson.** This is the second time on this project I've watched a silent-degradation bug get past me into a deployed surface (the first was M8's empty-ensemble → "NO TB" fail-open). The pattern is the same: a guard I assumed was tight (`if (!caps.hasHF) halt`) didn't cover the case I actually had (HF *configured* but the *model* dead). The fix in both cases was less about clever guard logic and more about making the failure visible — a verdict state that says "no reading", a per-provider status line that says *which* key is the problem, a banner that says "you haven't given me anything to work with". The ethos test isn't whether the guard *catches* the silent failure; it's whether the *user* can recover from it without my help. For a frontend-only BYOK app where every key path is the user's responsibility, that's the contract that matters most.

## Milestone 21 — Pivoting gpt-5.5 vision to PRIMARY perception (2026-05-24)

M20 made the structural problem explicit: both default HF heads I'd shipped since M1 (`Owos/tb-classifier`, `keremberke/yolov8m-chest-xray-classification`) returned `400 "Model not supported by provider hf-inference"`. M19 had also recorded a second gap — the local validated ONNX heads (`public/models/tb_head_t2.onnx` + `sequelae_head.onnx`, AUROC 0.922 LODO) cannot run in the browser today because they consume Rad-DINO patch tokens + TXRV logits that no in-browser pathway produces (Phase B gap). The deployed app no longer had a working primary perception. That's the situation M21 inherited.

**The decision and the trade I owned up to.** Promote `gpt-5.5` vision — already used for the quality gate, the VLM ensemble member, and the streamed adjudicator — to be THE primary perception, called via the Responses API's structured-output / json_schema mode. The trade is honest both ways: a general-purpose VLM is uncalibrated, hasn't seen the project's mimic stress test, and may overreact to scar/fibrosis/pleural-thickening (precisely the failure mode the sequelae head was meant to fix). It is also a step DOWN in measured accuracy from the M15 head (LODO AUROC 0.925). What it is — and the only reason this pivot ships — is a step UP in deployability: it actually runs on the live site today, where nothing else does.

**The schema, distilled from a hostile-critic GPT-architecture review.** I named the score `tb_score_uncalibrated`, not `tb_prob` — the field name is the safety contract. I named the scar field `scar_shape_score_uncalibrated`, not `s_inactive` — different model, different distribution, no shared threshold. I refused an "expert radiologist" persona prompt and a `chain_of_thought` field; the policy prompt is boring, constrained, "research preview, NOT diagnosis, output ONLY the schema, when uncertain choose ABSTAIN, no demographic priors, no inferred clinical history." Forced ABSTAIN rails fire deterministically *after* the call: `image_quality !== 'diagnostic'`, `projection === 'unknown'` with a positive screen, OR `confidence_band === 'low'`. The submission carries audit pins — prompt hash, schema version + hash, model id from the response envelope, image preprocessing version — recorded into every `PipelineRun.modelVersions` for the trail.

**Verifier call, not ensemble theater.** I deliberately did NOT keep the M4 self-consistency K=3 sampling. That was a calibration trick from when the VLM was the THIRD ensemble member; with the VLM as PRIMARY, three correlated self-samples look like ensembling but aren't. Instead: single deterministic primary call. ONLY when the result is borderline — `tb_score_uncalibrated ∈ [0.35, 0.65]` OR `confidence_band === 'low'` OR `mimic_features_present` contains a scar tag — fire ONE independent verifier call with a different-phrasing prompt under the same schema. If primary and verifier disagree on `tb_screen_result`, abstain. The audit pins record whether the verifier ran and whether it disagreed.

**Two separate safety nets, one careful boundary.** The M19 `applySequelaeEscalation` rule expects `s_inactive` from the local ONNX `InactiveSequelaeHead` and uses threshold 0.7126 derived from that head's own scar probe. The VLM's `scar_shape_score_uncalibrated` is NOT s_inactive — different model, different score distribution, no labeled validation. I built a SEPARATE function `applyVlmEscalation` in a SEPARATE file (`src/lib/pipeline/vlmEscalation.ts`) with a SEPARATE conservative starting threshold of 0.5 and a softer borderline floor (tb_score > 0.25, no upper cap — the VLM has no calibrated upper threshold to anchor against). The rule escalates on EITHER a scar-shape score over the heuristic threshold OR any scar-tag substring match. Same never-clear contract: NO_TB → ABSTAIN only; never overrides screen_positive; never demotes ABSTAIN. The unit tests pin the three required cases plus a unit test that asserts the VLM threshold is BELOW 0.7126 — if anyone ever copies the ONNX threshold into the VLM file, the test fails.

**Path-specific disclosures, not a one-size-fits-all footnote.** The M19 always-on scar-FPR disclosure described OUR trained head's NIH cross-site mimic FPR (~10% on Fibrosis / Pleural_Thickening). It's the wrong number to display when the actual perception was gpt-5.5 vision. I rebuilt VerdictCard's disclosure as path-specific: on the VLM path (today's deployed default), it reads — *"This result is produced by a general-purpose vision-language model (gpt-5.5 vision), not the project's validated Rad-DINO + TorchXRayVision research model. The TB score is uncalibrated and has not been validated to the LODO sensitivity/specificity numbers reported in the case study. The model may miss disease, hallucinate findings, overreact to scar/fibrosis/pleural-thickening, or abstain on image-quality issues. Research triage support only — not diagnosis."* — plus a smaller "Local validated ONNX heads (Rad-DINO + TorchXRayVision, AUROC 0.922 LODO) ship with this build but cannot execute without browser-side backbone features (Phase B gap)" note. The M19 disclosure still renders on the `onnx-primary` path when Phase B lands. A small `perception path: gpt-5.5-2026-04-23 (unvalidated VLM) · verifier ran · verifier disagreed` indicator appears under each verdict so the user can see which model produced the result, including whether the consistency check ran.

**The number I'd NOT inject into the prompt.** A reflexive way to anchor the VLM would be to feed it the LODO sensitivity/specificity numbers, the NIH fibrosis FPR, the scar Brier — "the project says expect X". I deliberately did not. Those numbers belong in the UI disclosure + the case study, where they are auditable, not in the inference prompt where they would invite the model to anchor on expected priors rather than read the image. The prompt knows nothing about our calibration story; that is the right boundary.

**Real numbers from the live verification.** I wired a one-off Node script (`scripts/verify-vlm-schema.mjs`) that reads the live `VITE_OPENAI_KEY`, posts our `submit_triage` schema to the Responses API on two real sample CXRs, and prints the parsed result + the model id from the envelope. Both calls returned HTTP 200 with strict-schema-validated JSON. The TB sample returned `screen_positive` with `tb_score_uncalibrated=0.72`, `confidence_band='medium'`, `scar_shape_score_uncalibrated=0.82`, and a mimic_features list including `apical_fibrosis`, `pleural_thickening`, `healed_scar`. The normal sample returned `screen_negative` with `tb_score_uncalibrated=0.04`, empty mimic list, but a `safety_flags: ['pediatric']` flag I didn't expect — useful, the kind of signal the M19 ONNX path couldn't surface. Both responses came back with `model="gpt-5.5-2026-04-23"`. Latencies were ~9s (normal) and ~12s (TB) on a single Vite-dev / Node call. The schema mode (`text.format = {type: 'json_schema', name, schema, strict: true}`) accepted our schema unchanged; no fallback to the `json_object` pattern was needed.

**What this verifies and what it does not.** A two-image hand-test verifies that the schema mode shapes load, the audit pins (model id, prompt hash, schema hash) record correctly, and the borderline + verifier wiring is structurally sound. It does NOT verify accuracy. The TB sample correctly returned screen_positive; the normal sample correctly returned screen_negative; that is a sanity check on TWO images. The case study still leads with the M15 head's measured LODO AUROC 0.925 as the project's headline accuracy claim — that number is from a real held-out evaluation against the project's own benchmark, and it is the number the VerdictCard's onnx-primary disclosure cites. The VLM-primary path has no comparable measured number. The disclosure says so out loud.

**The lesson.** This pivot is the second time I have watched a "validated and trained" pathway lose its runtime — first the HF backbones in M2 (CORS reality), now the in-browser feature-extraction in M19+M20. Each time the temptation is to declare the new path "good enough" by virtue of running. I tried hard to resist that here. The VLM path runs, and that is a real win — but the disclosure language, the path indicator, the separate-function-separate-threshold for the safety net, and the explicit "this is unvalidated" string in the audit trail are how I keep the win honest. The deployed app today is more deployable than M20 and less measured than M15. Both of those are true at once. The portfolio claim isn't "we hit 90% sensitivity" — never has been; the WHO bar takes labeled data we don't have — it is "we built the system so an honest reader can tell, from the UI alone, which model produced this verdict and what its caveats are."

**Test + build state.** vitest **86 passing** across **9 files** (calibration 11 + sequelaeEscalation 10 + VerdictCard 14 + NoKeysBanner 2 + ProviderStatusBadge 7 + providerStatus 8 + **vlmTriage 22 + vlmEscalation 12**). `npm run build` clean. Training tests untouched. Branch `feat/vlm-primary-orch` NOT merged to main per the brief.

## Milestone 22 — LOCAL-MODE TRIAGE: actually run the validated pipeline (2026-05-24)

For four milestones I have been telling the case study that the project's validated head — Rad-DINO + TorchXRayVision + the deployed `TBHeadT2` + the `InactiveSequelaeHead`, AUROC 0.922 LODO on 13,092 held-out predictions — exists, has measured numbers, and sits in `data/tb_head_t2.pt` and `data/tb_head_inactive.pt`. And for four milestones the deployed app has not run it. M19 exported it to ONNX and admitted the Phase B browser-side feature pathway was the gap. M20 confirmed the M1 HF backbones were dead. M21 pivoted to gpt-5.5 vision as the primary perception because the alternative was no perception at all. M22 is the milestone where I stop hiding behind that and use my own machine: my M4 has the weights, has MPS, has the HF cache. The validated pipeline can run **today**, locally, on the user's box. The deployed Netlify app stays on the M21 VLM-primary path (it has no way to reach a user's localhost), and the LOCAL-mode user gets the calibrated head as the primary.

**The shape that earned the name "local-onnx-via-server".** A single FastAPI server (`training/server.py`) wraps `training/triage_core.TriageEngine`, which is the SINGLE-SOURCE-OF-TRUTH module the CLI (`training/triage_cli.py`) and the orchestrator (transitively, via `src/lib/providers/localTriage.ts`) both call. Engine loads once on startup — Rad-DINO ViT to MPS, TorchXRayVision DenseNet on CPU with `op_threshs=None`, the deployed `tb_head_t2.pt` weights, the `tb_head_inactive.pt` weights, the lung segmenter — then per-request inference is sub-second. CORS is narrow: `localhost:5173` and `127.0.0.1:5173`, not `*` (verified with an `Origin: http://evil.com` curl that the server rejects with 400 and no allow-origin header). The path indicator in `Adjudication.perception_path` gets a NEW value, `'local-onnx-via-server'`; I extended the existing union rather than parallel it, so VerdictCard's path-specific switch has the right place to hang a disclosure.

**The discipline I kept honest with myself was the preprocessing.** The M12 audit traced our biggest accuracy gains to a specific, tested preprocessor: percentile-clip + min-max + antialiased resize at WORKING_RES=1024, lung-mask soft-crop with a 4% dilate + 18% margin, then a *letterbox-to-square* (so Rad-DINO's `shortest_edge=518 → center_crop=518×518` becomes a true uniform resize and no apical or costophrenic content gets chopped). Any deviation silently degrades accuracy because the encoder is FROZEN — it has to see images close to its training distribution. I refused to paraphrase that math here. `triage_core.py` imports `_harmonize`, `letterbox_to_square`, `PATCH_GRID`, `_detect_inversion`, and `_read_gray` *directly* from `extract_features.py` and the lung segmenter via `preprocess._get_seg`. The seg-and-letterbox single-image path is the only chunk of new procedural code, and it's a straight batch-size-1 port of `extract_features.seg_crop` line-for-line, with comments saying so. Calibration constants are READ from `tb_threshold_t2.json` (T=1.5914955557015147, thr_at_95sens=0.6104530692100525) and `tb_inactive_meta.json` (T_sequelae=1.1312716480691851) on engine load — never pasted into source. If a future re-fit propagates new T's to the JSON, the engine and the test tripwire pick them up automatically.

**The numbers I measured on my M4.** Two CXRs — `public/samples/tb-sample-1.jpg` (a real TB film) and `public/samples/normal-sample-1.jpg` (a healthy chest) — through the local pipeline:
- **tb-sample-1.jpg**: calibrated `tb_prob = 0.9999769` (logit +16.99 / T=1.5915), `s_inactive = 0.0463` (under T_sequelae=1.1313), verdict **TB**, safety net not fired. Warm latency: harmonize 6 ms, seg 116 ms, rad_dino 117 ms, txrv 62 ms, heads 4 ms, **total 317 ms**.
- **normal-sample-1.jpg**: `tb_prob = 0.0034`, `s_inactive = 0.4956`, verdict **NO_TB**, safety net not fired. Warm latency: harmonize 6 ms, seg 119 ms, rad_dino 120 ms, txrv 67 ms, heads 4 ms, **total 292 ms**.
Both samples were the *same* two M21 verified against gpt-5.5 vision (where the VLM had returned uncalibrated `tb_score_uncalibrated=0.72` for TB and `0.04` for normal in 9–12 seconds per call). The local pipeline is on the same order of accuracy on these two films AND ~30× faster AND — the load-bearing one — its number IS calibrated. The 0.9999/0.0034 are real probabilities under the head's fitted temperature, not vibes-on-the-VLM-bar.

**Two sanity checks I owed myself.** Determinism: same bytes in twice produced bit-identical `tb_prob` to 6 decimals across runs (MPS fp32 noise was below 1e-7 in practice; my test asserts within ±0.001 to leave room). Server vs CLI parity: `curl -F file=@tb-sample-1.jpg http://localhost:8000/triage | jq .tb_prob` and `triage_cli.py --json` both returned 0.999976910628603 to the last digit. I wrote a deterministic Python test (`training/test_triage_engine.py`) that asserts the calibrated `tb_prob` on `tb-sample-1.jpg` stays within ±0.02 of the M22 baseline number across runs — if a future change to the preprocessing, the head, or the calibration JSON drifts that number more than 2 percentage points on this committed sample, the test fails. It's the same M22 EXPERIMENT_LOG row's number, made into a tripwire.

**The orchestrator pivot, named in one sentence.** When `settings.localMode === true` AND the local /triage server is reachable, the local validated pipeline IS the primary perception, the validated thr_at_95sens=0.6105 IS the threshold the verdict is decided at, the s_inactive feeds the M19 sequelae escalation rule, and gpt-5.5 vision is reduced to a borderline second opinion that fires only when `tb_prob ∈ [0.35, 0.65]` OR `s_inactive ≥ 0.7126` — and only forces ABSTAIN on disagreement, never overrides a confident verdict. When local mode is off OR the server is unreachable, the M21 VLM-primary path runs unchanged, banner and all. That's it. The whole change to `runPipeline` is a single block before the M21 stages, with explicit fall-through on local error. No parallel shape, no shadow orchestrator.

**Path-specific UI disclosure, third time I've had to add one.** VerdictCard already had two disclosures: the M19 onnx-primary one ("Higher false-positive rate ~10% on scar-shaped findings") and the M21 vlm-primary one ("This result is produced by a general-purpose vision-language model, not the project's validated research model"). The local-onnx-via-server disclosure replaces both with the right honest copy: *"This result is produced by the validated Rad-DINO + TorchXRayVision research model running on your machine. Reported LODO sensitivity 0.800 / specificity 0.911 / AUROC 0.922 on 13,092 held-out predictions; per-site recalibration recommended. Higher false-positive rate (~10%) expected on radiographically scar-shaped findings (healed fibrosis, pleural thickening — M18 NIH stress). Not a medical device."* — and a path indicator that says `perception path: local trained model (validated) · gpt verifier ran/disagreed/not fired`. The "general-purpose VLM" disclosure is *suppressed* on this path; it would be wrong to display it when the actual perception was our head.

**What surprised me about M4 perf.** I had budgeted "sub-second" as an aspiration; warm runs landed at 292–317 ms total, with the lung-mask segmenter (118 ms) being the dominant stage — bigger than Rad-DINO itself (117 ms on MPS) and the small heads (4 ms). The TorchXRayVision DenseNet on CPU was 62 ms, comparable to Rad-DINO on MPS. This says two things: the M4's neural engine isn't the bottleneck, the segmenter is; and the in-browser ONNX-runtime fantasy I had for M19 wasn't crazy on a per-image basis, but it was the *segmenter* and the harmonize+letterbox preprocessing that were silently making it expensive — getting those right is more important than getting the heads onto WebGL. I am now glad I deferred Phase B and went to local server.

**What the architectural detail that mattered most was.** I tried very hard to write `triage_core.py` so it could not silently diverge from training. The preprocessing import line — `from extract_features import _harmonize, letterbox_to_square, PATCH_GRID, ...` — is the single most load-bearing line in the whole milestone. If I had reimplemented even one of those (the percentile clip, the antialiased resize, the symmetric letterbox), the calibration would have been silently wrong by some small amount, the verdict would still have looked "fine" on most images, and the EXPERIMENT_LOG tripwire would not have fired on the headline sample (because the deviation would be image-dependent). The discipline of NEVER paraphrasing what you have already tested is, when frozen-encoder preprocessing matters this much, the difference between a "we use the validated head" claim and an actually-validated head.

**The honesty I have to keep.** The local pipeline running ≠ a 95%-sensitivity device. It IS the validated head, the validated threshold, and the validated calibration — but those numbers were measured against *radiographic* labels in a LODO design, and per the M9 panel and the CLAUDE.md ethos, they remain a research-preview number, not a deployment claim. The local-mode disclosure says LODO 0.800 / 0.911 / 0.922 *and* "per-site recalibration recommended" *and* the M18 ~10% mimic FPR *and* "Not a medical device" — all true, all of them load-bearing. The reason M22 is a satisfying milestone is not "we hit 95% sensitivity" — never has been, never will be on this dataset — it is "we are now running the head we measured, on the path the case study claimed, with the path indicator and the disclosure that tell the user exactly which model produced this verdict." The portfolio claim has not moved. The infrastructure to honestly carry the claim has.

**Test + build state.** vitest 86 → **99 passing** across **10 files** (calibration 11 + sequelaeEscalation 10 + VerdictCard 14 + NoKeysBanner 2 + ProviderStatusBadge 7 + providerStatus 8 + vlmTriage 22 + vlmEscalation 12 + **localTriage 6 + orchestrator 7**). Training tests **stress_metrics 6/6 + onnx_parity 3/3 + new test_triage_engine 6/6 = 15 total**. `npm run build` clean. New venv deps (added once: `fastapi`, `uvicorn[standard]`, `python-multipart`, `openai`). Branch `feat/local-mode-triage` NOT merged to main per the brief.

## Milestone 23 — removing Hugging Face from the runtime path (2026-05-25)

By M23 I had been narrating around a runtime dependency that hadn't worked for months. The free `hf-inference` router stopped hosting our two default classifiers (`Owos/tb-classifier`, `keremberke/yolov8m-chest-xray-classification`) in 2024-2025 — verified live in M20 with a `HTTP 400 "Model not supported by provider hf-inference"` body for both. Even the backbones we use offline (`microsoft/rad-dino`, `torchxrayvision/...`) are not deployed on the router for serverless inference. M21 worked around it by promoting gpt-5.5 vision to the primary perception. M22 built the local FastAPI path so the validated 0.922-AUROC head could actually run. By M23 the HF leg of the orchestrator was firing into the void on every request: red errors in the trace, two stages reporting `error` in the UI, and a settings drawer asking for an `hf_...` token whose only effect on the deployed app was to make those errors slightly more polite.

So I cut it. Not refactored, not feature-flagged — deleted. The HF provider client, the `Settings.hfToken` field, the HF status badge case, the `tbClassifierHf` / `generalCxrHf` / `embeddingEndpointUrl` overrides, the `HfError` class, the `'hf'` member of the `Provider` union, the `'hf-ensemble'` member of `perception_path`, the legacy ensemble auxiliary stages in the orchestrator — all gone. The `~/.cache/huggingface/` directory and the HF Python library stay because that's how `training/triage_core.py` reads Rad-DINO and TXRV weights into the engine at startup with `HF_HUB_OFFLINE=1`. That is a file-system load, not a runtime API call, and the frontend has no awareness of it.

**The simplification I let happen.** Once HF was removed, the "ensemble" was a single member dressed up in EnsembleResult clothing — there's only ever one perception path now (local-triage when on and reachable, otherwise gpt-5.5 vision). I considered preserving the multi-member shape for a future BYO Replicate slot, but I picked honesty: the orchestrator now wires a single-member `EnsembleResult` and the AgentTrace UI shows two cards (`Local validated head` when the local-mode run produced a member; always `GPT-5.5 Vision`) instead of the four it used to show. The synthetic ensemble shape stays so the existing UI doesn't have to be re-architected, and `weightedScore` / `std` / `disagreement` still get computed — they just collapse to trivial values when there's one member.

**What's left, named honestly.** Two perception paths: local-mode primary (the validated, calibrated head on the user's machine) and gpt-5.5 vision primary (the uncalibrated VLM for the deployed Netlify app). Replicate stays as a BYO classifier slot — empty by default, surfaces an honest "not configured" state when not set up. The Settings drawer lost three fields (HF token + two HF model overrides + the HF embedding endpoint URL) and the model-overrides panel is now half its previous size. The `NoKeysBanner` still appears when no key is set, but it now also stays silent when local mode is on, because local mode is itself a valid "configured" state.

**Vitest count after.** 99 → 99. Most of the HF-shaped test rows were rewritten to assert the same diagnostic behavior on the Replicate path (which inherits the same `classifyHttpFailure` dispatcher). The two HF-specific cases in `ProviderStatusBadge.test.tsx` were replaced by Replicate equivalents; the orchestrator test got a new assertion that the local member's `provider_used` is now `'local-triage'` rather than the legacy `'hf'` pseudo-tag; the `VerdictCard` "perception unavailable" copy was updated and its test follows. `npm run build` stays clean under strict TS, no `any`. Training tests untouched.

**What I learned.** When a dependency stops working and the replacement is strictly better, the right move is deletion not patching. The "Three providers called directly from the browser" framing I'd been carrying since M1 had become a story I was telling, not a fact about the code — and the story was leaking through to the user as visible errors. The case study is the place to record what I built; the running app is the place to record what works. M23 brought those two back into alignment.

## Milestone 24 — surfacing the validated model's intermediates, and the diagnostic that came out of it (2026-05-25)

Through M22 the local-mode pathway was producing a calibrated `tb_prob`, a calibrated `s_inactive`, and a deterministic verdict — and then the engine threw away every intermediate that had gone into making that verdict. The 8x8 box-evidence map from the BoxEvidence head's LSE-LBA pool, the per-zone calibrated probabilities from the ZonalSoftOR (7 zones — upper/mid/lower per hemithorax + hilar/mediastinum), the 18 TorchXRayVision named-finding probabilities the fusion head's lever consumed — all of it computed, all of it discarded, all of it visible in the `TriageResult` if I had only chosen to surface it. M24 was about surfacing it. Three goals, in order of safety contract: (1) extend `LocalTriageResult` with these intermediates as OPTIONAL fields so older servers and the VLM-primary path degrade gracefully; (2) build four UI components that render them honestly in the existing `VerdictCard`; (3) add a gpt-as-interpreter provider that TRANSLATES the validated model's structured output into a clinician-style paragraph WITHOUT being allowed to disagree with the verdict.

**The intermediates and the contract that protects them.** I extended `training/triage_core.py`'s `TriageResult` to carry `box_evidence_grid` (the row-major 8x8 sigmoid of the per-cell box scores), `zonal_scores` (the 7 calibrated per-zone TB probabilities under T), `txrv_pathologies` (the 18 named findings sigmoid'd for human eyes — raw logits feed the head, sigmoid'd values feed the UI), `crop_box` (the seg-driven letterbox in original-image pixel coords), and `inversion_detected`. The wire shape mirrors that on the TypeScript side in `src/lib/providers/localTriage.ts`, every field individually optional and the validator non-rejecting if any is missing. `Adjudication.local_enrichment` extends (not parallels) the existing Adjudication shape; the VLM-primary path never populates it; UI components render only the sub-fields that exist.

The honest framing per panel: `BoxEvidenceHeatmap` says explicitly *"NOT a radiologist annotation; not a region of interest in the clinical sense"* (the BoxEvidence head's LSE-LBA pool target is a soft attention mass, not a clinical ROI, and the panel must not pretend otherwise — Li et al. 1803.07703's framing). `PathologyList` says *"feature inputs to the TB head, not independent diagnoses"* (the TXRV DenseNet was trained on a 14-finding label set and we use its raw logits as fusion inputs, not as diagnoses). `ZonalBars` uses the HONEST 7-zone schema from the trained ZonalSoftOR — I refused the brief's suggested 8-zone split because the trained model has `N_ZONES=7` and there is no L/R split of the hilar/mediastinum channel; making one up would be a UI fabrication. `ClinicianReport` (the gpt-as-interpreter panel) carries the disclosure *"Generated by gpt-5.5-2026-04-23 vision from the validated model's structured output (schema gpt-interpreter-v1; confidence band X; latency Ys). Narrative interpretation only — does not change the verdict."* That last sentence is the M24 honesty contract: the verdict came from the validated head and gpt is allowed to translate, not to overrule.

**The gpt-as-interpreter provider, and the keep-AUROC-load-bearing prompt design.** `src/lib/providers/gptInterpreter.ts` is the opposite of M21's `vlmTriage`. M21 asks gpt-5.5 vision to give a verdict because no in-browser perception backbone exists. M24's interpreter asks the same model to TRANSLATE the validated pipeline's structured output. The prompt forbids the model from disagreeing with the verdict, forbids it from inventing zone keys not in the evidence message, and forbids chain-of-thought scaffolding (which CoT-in-radiology-VLMs research has shown invents findings). The evidence message pins the model: it sees the verdict, the calibrated `tb_prob` under T, the threshold, `s_inactive` under T_sequelae, the top-5 zonal scores and the top-5 TXRV findings (not the full grid — the model would invent grid-cell labels we cannot back). The schema returns five fields: `finding` (1-2 sentences), `key_regions` (filtered to the subset of zone names we passed in — a hard guard against fabrication), `confidence_qualifier`, `top_differential_alternatives` (each grounded in TXRV or scar score, not in pixels), and `refusal_or_limitation`. Audit pins (prompt_hash, schema_version, schema_hash, model_id_from_response) on every call so an exported run can join a narrative to the exact contract that produced it. Cost ~$0.01-0.04 per call, latency ~5-15s, lazy CTA so the user opts in.

**The diagnostic that wrote itself once the intermediates were visible.** I ran the 23-image blind set through the enriched engine (`training/test_blind_set_enriched.py`, 11 TB+ from public clinical sources + 12 NIH No_Finding labels). The baseline was honest and unflattering: **5/11 TB+ correctly called TB, 6/11 missed as NO_TB; 12/12 NO_TB correctly cleared.** Two of the misses (#1 tb_blind_01 tb_prob=0.60, #4 tb_blind_04 tb_prob=0.48) sat in the verifier band and would route to gpt-5.5 vision as a borderline second opinion. Four (#6, #7, #9, #10) returned `tb_prob` essentially zero and would clear confidently. And it was those four where the intermediates rewrote the story.

Every one of the four confident misses had the same signature: `box_evidence_max ≥ 0.882`, `TXRV Lung_Opacity ≥ 0.44`, per-zone scores ≈ 0 across all 7 zones, `tb_prob < 0.05`. Worked through case by case in the ASCII heatmaps (rendered as `.` <0.05 `o` <0.20 `O` <0.50 `#` >=0.50): #6 had a 4-cell red-hot region in the mid-left lung; #7 had a 6-cell region across mid-left and mid-right; #9 had nearly the entire mid-zone lit up; #10 had a strong mid-right cluster. The BoxEvidence head was firing CORRECTLY on the pathology. The TXRV DenseNet was reporting CORRECTLY that there was lung opacity. And the zonal+fusion heads — the actual TB-classifier decision layer — were refusing to call it TB because the pattern did not match what they had been trained to recognize.

The conclusion writes itself: **the validated head has learned a too-narrow definition of "TB-pattern".** Specifically, the apical/cavitary parenchymal morphology that dominates TBX11K + TB-Portals. When real TB presents with mid-zone consolidation or pleural effusion, the perception-layer evidence channels (BoxEvidence + TXRV) correctly identify pathology, but the classification heads refuse to call it TB. This is an **atypical-presentation failure**, not a preprocessing failure (the heatmap shows correct localization), not a calibration failure (the calibrated `tb_prob` is in the same range as confident NO_TB controls), and not a domain shift (these are mid-zone consolidative TB images, the most common South-Asian smear-positive morphology). It is precisely the rate-limiting failure mode that the LoRA fine-tune (training task #37) was positioned to fix — and now I have a 4-case catch set to validate against the day that fine-tune lands.

**What I learned.** Surfacing intermediates is not a UX flourish — it is a debugging discipline. For four milestones the validated head had been producing `verdict: NO_TB` and I had been believing it because the calibrated number said so. The 8x8 grid says the model saw it. The TXRV score says the backbone saw it. The zonal head says it didn't recognize it as TB. Three signals about the same image, three honest answers, and the verdict had been quoting only one of them. The M24 evidence panels are the bridge between the model's internal honesty and the verdict's external claim — and the diagnostic they enabled is the cleanest case I have for what the roadmap should fund next.

## Milestone 26 — the asymmetric-evidence ABSTAIN rule, justified by what M24 surfaced (2026-05-25)

M24 showed the model: it sees pathology, doesn't call TB. M26 wrote the deterministic safety net that says when that happens, route the case to ABSTAIN ("model sees pathology in lung region but does not match learned TB-pattern — second-opinion required") rather than confidently clear. Third NO_TB→ABSTAIN escalator, alongside `applySequelaeEscalation` (M19, scar shape) and `applyVlmEscalation` (M21, VLM scar heuristic). All three are pure functions composed in `combineLocalIntoAdjudication` with the same one-way contract: push NO_TB up to ABSTAIN only, NEVER demote TB or ABSTAIN, NEVER override a confidently positive call.

**The rule, and the threshold derivation that anchors it on data.** When the local pipeline returns `verdict: no_tb` AND `tb_prob < TB_PROB_LOW_THRESHOLD` AND `box_evidence_max >= BOX_EVIDENCE_HIGH_THRESHOLD` AND `top_pathology_score >= PATHOLOGY_HIGH_THRESHOLD` (the max of the five TB-relevant TXRV labels — Lung_Opacity, Effusion, Lung_Lesion, Infiltration, Consolidation), escalate to ABSTAIN. Three thresholds, every one derived on data, not chosen by feel.

I swept candidate triples on the 23-image blind set diagnostic (`(tb_low ∈ [0.05, 0.24], box_high ∈ [0.50, 0.95], path_high ∈ [0.20, 0.59])` at 0.01 resolution). The catch set is the 4 confident TB-misses #6, #7, #9, #10. Their minimums are box_max=0.882 (case #7) and top_pathology=0.44 (case #10) — these are the data-anchored floors. I rounded box DOWN to 0.88 so the rule isn't brittle to a 0.001 wobble in a future run; pathology stays at 0.44 (the exact floor of the catch set). For tb_low I picked 0.20 — the catch set covers tb_prob ∈ [0.003, 0.040] so anything in [0.04, 0.20] gives the same 4/4 catch on this set, but 0.20 (a) aligns with the M26 VLM verifier band widening (also 0.20, see below) and (b) keeps a small buffer above the catch set so a near-borderline mid-zone TB at tb_prob=0.15 still escalates. Honestly-unmeasured: per-site performance, deployment prevalence, whether path_high=0.44 holds on a non-public TB cohort. The rule is advisory, not a calibrated threshold, and the header comment in `asymmetricEvidence.ts` says exactly that.

The 13k LODO cache (`data/image_oof_logits.npz`) gave a useful sanity check on the tb_low value alone — ~13% of TB+ have `tb_prob < 0.20` (the slice the catch set is sampling from), and ~84% of TB- do. Without the box+pathology AND-gate, forcing ABSTAIN on `tb_prob < 0.20` would abstain on 9,512 of 11,345 true negatives — disastrous. The joint rule is what makes it tractable: box AND pathology AND low-tb_prob all have to fire, and the AND-gate is what cuts the false-abstain rate down to something we can actually ship.

**The NIH #18 label-noise quirk, documented as expected behavior.** One NO_TB case — `notb_blind_07` (NIH No_Finding label) — trips the rule with `tb_prob=0.003, box_max=0.998, Lung_Opacity=0.83`. The NIH label says No_Finding; the radiograph carries visible pathology; the model correctly saw it but didn't call TB. Under the rule the case gets routed to ABSTAIN ("model saw something, get a second opinion"). That is correct design behavior, not a defect — and the rule's header comment + the unit test pin it explicitly so a future reader doesn't try to "fix" it by special-casing the file.

**Applied to the 23-image blind set, the headline numbers.** Before M26: 5/11 TB+ correctly TB, 6/11 missed as NO_TB. After M26 the four confident misses (#6, #7, #9, #10) all reroute to ABSTAIN; the two borderline misses (#1, #4) remain NO_TB at the local-pathway base verdict (the M26 verifier-band widening from 0.35 to 0.20 means they fire the gpt-5.5 vision verifier as a consistency check; the verifier's own disagreement decision is independent of this rule). The new headline: **sensitivity-with-ABSTAIN-counted-as-rescued = (5 TP + 4 ABSTAIN) / 11 = 0.818** (target was ≥ 8/11; got 9/11). **Specificity-via-clear = 11 / 12 = 0.917** (the 1/12 false-abstain is the NIH #18 label-noise case). Strict sensitivity stays at 5/11 because abstain doesn't count as TP — and the strict number is the one the user-facing disclosure should say.

**The borderline-low widening, and the cost-benefit it trades.** The M24 diagnostic also surfaced two TB cases at tb_prob 0.48 and 0.60 that the local pipeline read as NO_TB. Under the M22 verifier band [0.35, 0.65] both already fire the gpt-5.5 vision verifier consistency check — but a wider band catches more cases. I lowered `VLM_BORDERLINE_LOW` from 0.35 to 0.20 and made the same change to `LOCAL_BORDERLINE_LOW` in the orchestrator's local-mode GPT-verifier trigger. The new band [0.20, 0.65] catches uncertain cases the box+pathology AND-gate of the asymmetric-evidence rule wouldn't catch — cases where the model said "low but not zero" without strong box or pathology evidence. The cost: a small bump in gpt-5.5 vision verifier calls per session (~7% of LODO cases sit in the new [0.20, 0.35] slice). I think it's worth it. The brief is a research preview where catching missed TB is what the safety net is for.

**Vitest after, build after.** Vitest 130 → 159 across 16 files (added asymmetric-evidence rule 20 + 4 evidence-panel files BoxEvidenceHeatmap 5 + PathologyList 5 + ZonalBars 5 + ClinicianReport 8 + the new vlmTriage band-edge case + the PathologyList chip-count regex fix). `npm run build` clean under strict TS no `any`. Training tests untouched (6+3+6+6 = 21). Branch `feat/local-model-enrichment` carries 9 commits: the original 3 from the interrupted M24 session (intermediates + types + gpt-interpreter) plus 6 new commits (BoxEvidenceHeatmap + PathologyList/ZonalBars + ClinicianReport + VerdictCard integration + asymmetric-evidence rule + orchestrator wiring + verifier widening).

**What I learned.** A safety net rule earns its thresholds. I picked 0.88 not because it sounded right but because the minimum box_evidence_max in the catch set is 0.882. I picked 0.44 not because it sounded right but because case #10's Lung_Opacity is 0.44. The threshold-derivation comment in `asymmetricEvidence.ts` is six paragraphs long because every number has a paper trail back to a measured value — and the day someone re-fits this rule on a different cohort, that paper trail is the only thing that tells them which floor moved and which floor held. The other lesson: the rule does not pretend to fix the model. The model still has a too-narrow TB-pattern definition. The rule's job is to refuse to confidently clear a case the model has visibly seen pathology in, while we wait for the LoRA fine-tune to widen the definition. ABSTAIN is the honest output; confident NO_TB on a clearly-pathological film is not.

## Milestone 27a — subgroup fairness audit on the M26 abstain rule (2026-05-25)

The arXiv 2010.14134 paper ("Selective Classification Can Magnify Disparities Across Groups", Jones et al.) shows that learning-to-abstain rules can amplify per-subgroup disparities — exactly the failure mode that hurts under-represented populations with atypical-TB morphology (HIV+, elderly, immunocompromised, post-tubercular). M26's rule, by design, fires more often on radiographs that look like TB-mimics — that is its job. The fairness question the literature forces me to ask is whether that elevation is PROPORTIONAL to the catch benefit, or whether one subgroup gets disproportionately routed to ABSTAIN with no rescue benefit (a "Clever-Hans for ABSTAIN"). I built the audit to answer that.

**The honest scope, written first because it bounds every number that follows.** The 13k LODO cache at `data/image_oof_logits.npz` carries only `image_logit`, `label`, and `source` — the box-evidence map and the TXRV pathology block that the M26 rule reasons on were NOT cached during the original LODO run (which pre-dated M24 enrichment). A strict held-out OOF fairness audit on the full 13k therefore requires re-running LODO end-to-end with the M24-enriched outputs cached. I did not do that. Instead, Dimension 1 of this audit runs the deployed T2 head on `data/features.npz` (the same 13k training-set features that DO carry the inputs the rule needs) — that is a TRAINING-SET probe, NOT a held-out probe, and positives will correctly classify (the head saw them in training). The per-source numbers it produces characterize the rule's behavior on the deployment-distribution INPUTS, not on out-of-fold predictions. Dimension 2 has no such limitation — NIH 10k was never in training data, so the per-finding ABSTAIN rates on NIH are a genuine off-distribution fairness probe. Recording the scope limit here so a later reader doesn't read more into Dimension 1 than the data supports.

**The audit, the gates, and the literal verdict.** New `training/fairness_audit_m27a.py` mirrors the M26 rule's three thresholds (`TB_PROB_LOW_THRESHOLD=0.20`, `BOX_EVIDENCE_HIGH_THRESHOLD=0.88`, `PATHOLOGY_HIGH_THRESHOLD=0.44`) and the five TB-relevant TXRV labels VERBATIM from `src/lib/pipeline/asymmetricEvidence.ts`. A test in `training/test_subgroup_fairness.py` is the drift tripwire: it greps the canonical TypeScript file and fails the run if any constant moves without updating both sides. The audit produces `data/m27a_fairness_audit.json` with three dimensions: per-source (Dim 1, training-set scope as above), per-finding on NIH (Dim 2, 15 subgroups), and the catch-vs-cost ratio per subgroup with n≥10. The gates: PASS = no subgroup above 3x baseline AND all positive-containing source subgroups have catch/cost ≥ 1.0; WATCH = 1-2 subgroups in 3-5x range OR a source subgroup with catch/cost < 1.0; FAIL = any subgroup > 5x baseline with zero catch (the Clever-Hans for ABSTAIN signature). I did not fudge the gates — they are constants at the top of the audit module.

**Dimension 2 (NIH per-finding) — the headline that matters for fairness.** Baseline (NIH No_Finding ABSTAIN rate) = **8.21%** (475/5788). Ranked descending:

The TBX11K rate is the one that means anything: 8.74% of TBX11K's 7,439 negatives abstain, dominated by the same scar-mimic + atypical-morphology subset M16's sequelae work already characterized. This is consistent with the deployment intuition: on a population dominated by training-distribution negatives, the rule's cost is the false-abstain rate on the radiographic-mimic subset.

**The literal gating verdict is FAIL — and the WHY-IT-FAILS reading is what matters.** Two NIH subgroups (Edema 5.92x and Consolidation 5.37x) cross the 5x threshold with catch=0 (NIH has no TB positives, so catch is zero by construction for every NIH subgroup). I am NOT going to fudge the gate. But the honest reading is: those two findings are EXACTLY the rule's catch set by design. The M26 catch set was four mid-zone consolidative TB cases with `box_evidence_max ≥ 0.882` and `Lung_Opacity ≥ 0.44` — the radiographic morphology of "consolidative pneumonia-like findings on a film without learned TB-pattern." Edema and Consolidation in NIH look exactly like that to the perception backbone: a TXRV `Lung_Opacity` mean of 0.91 and 0.88 respectively (vs 0.68 on No_Finding), with `box_evidence_max` mean 0.91 and 0.87. The rule is firing on the radiographic signature it was tuned to fire on. It is doing its job.

**The fairness-relevant question my gate-as-written does not answer.** The literature's concern is demographic disparity — a rule that abstains disproportionately on a protected class without rescue benefit. NIH per-finding labels are radiographic findings, not demographic groups. To turn the audit into a true demographic-disparity check we would need NIH metadata stratified by age band / sex / view position / scanner — the dataset's `view_position` column carries one of these signals but the others are not in `Data_Entry_2017.csv` in the form the literature requires. I am recording this as the audit's NEXT step, not pretending the current audit covers it. What the current audit DOES tell me clearly: the rule's elevated-firing classes are the ones it was designed to catch (mid-zone consolidative pathology, pleural effusion, mass — the atypical-TB morphology), and Fibrosis — the canonical healed-scar mimic — actually sits at **1.99x baseline (BELOW the 3x watch line)**, which is honestly surprising and informative. The rule does not preferentially fire on healed-TB sequelae shape; it fires on active-disease pathology shape. M16's `s_inactive` head and M19's sequelae escalator are the layer responsible for the scar-mimic case; the M26 rule is the layer responsible for the consolidative-atypical-TB case. They are doing different jobs and the audit confirms it.

**One more honest number.** The 8.21% No_Finding baseline ABSTAIN rate on NIH is higher than I expected. For a population that is supposed to be the "easy negative" subgroup, 1 in 12 abstaining is a non-trivial review burden. The mean `box_evidence_max` on NIH No_Finding is 0.68 and mean `pathology_max` is 0.22 — neither alone trips the rule, but the right tail (cases where both signals happen to be high on a film NIH labeled "no finding") is the source. M18's NIH stress test already showed the deployed head has a ~10% confidently-wrong rate on the Fibrosis/Pleural_Thickening subset; the 8.21% baseline ABSTAIN tells the same story from the rule's side — when the backbone evidence channels disagree with the classifier on an off-distribution image, the rule routes to ABSTAIN. That is the conservative-by-design behavior. The deployment-prevalence read: at a 1% TB prevalence, an 8.21% baseline ABSTAIN means roughly 1 in 12 negatives goes to review. The cost is real and the disclosure should say so.

**Verdict (recording, not editing).** Literal gate result: **FAIL** on the 5x-with-no-catch threshold, with two subgroups (NIH Edema and NIH Consolidation) tripping. The literal verdict is honest because the gate is honest — those subgroups DO get >5x the ABSTAIN rate and they DO have zero catch (NIH has no TB+ to catch). The contextual verdict is that the elevated-firing classes are the rule's literal design targets (mid-zone consolidative + pleural pathology + atypical-TB morphology), the canonical healed-scar mimic (Fibrosis) sits below the 3x watch line, and the audit lacks the demographic stratification a true Jones-et-al-style fairness check needs. **Decision: keep the rule, document the elevated NIH firing pattern in the UI disclosure ("the rule will route consolidative pathology to ABSTAIN at a rate ~5x the no-finding baseline — that is the rule's catch target, but it does increase the review burden on consolidative-pathology films"), and re-run the audit with NIH demographic metadata + a held-out OOF cache once those land.** The audit's JSON is at `data/m27a_fairness_audit.json` for the next drift-log row.

**Tests.** New `training/test_subgroup_fairness.py` carries 9 tests: (1) a drift tripwire that pins the audit's rule constants to `src/lib/pipeline/asymmetricEvidence.ts` (greps the canonical file); (2) TXRV label-to-index round-trip; (3) the AND-gate behavior of `_abstain_mask` (each condition individually load-bearing); (4) the forward pass returns finite probabilities in [0,1]; (5) per-finding rows = FINDING_COLS in order with a sane baseline (No_Finding's multiplier is exactly 1.0); (6) per-source emits one row per training source; (7-8) synthetic PASS and FAIL scenarios for `gating_verdict`; (9) end-to-end `run_audit()` writes the expected JSON keys. Training tests now 21 + 9 = **30 passing**. Frontend untouched (the audit is offline-only); vitest still 159; `npm run build` clean.

**What I learned.** A fairness audit reports back the rule, not the world. The arXiv paper's framing is correct in the abstract: abstention CAN amplify disparities. Whether it DOES in any given system depends on (a) what dimensions you audit along and (b) what gate you choose. Auditing along NIH radiographic findings (the dimensions I had data for) tells me whether the rule fires preferentially on radiographic mimics; it does, exactly as designed. Auditing along demographics (the dimension fairness genuinely cares about) needs metadata I don't have here. The honest output is to record both halves: what the audit DOES show (rule behaves as designed) and what it does NOT show (no demographic stratification, no held-out LODO scope), and to gate the verdict on both. The other lesson: gates that produce a clean literal FAIL are MORE useful than gates that always produce a comfortable PASS. The audit's FAIL is the catch-zero-because-NIH-has-no-positives edge case — and reading that edge case carefully forced me to write down what the audit was actually measuring, which is what the case-study reader needs.

## Milestone 27b — A/B testing rad-dino-maira-2 as a drop-in backbone (2026-05-25)

The M24 diagnostic had four confident TB-misses on the Cohen blind set — `tb_blind_06, 07, 09, 10` — all with `box_evidence_max ≥ 0.882` and TXRV `Lung_Opacity ≥ 0.44` but `tb_prob < 0.05`. The model was seeing pathology and refusing to call it TB. M26 wrote the deterministic ABSTAIN rule that catches three of those four cases. Task #37 (a LoRA fine-tune of the perception backbone) is the open lever for actually closing the gap rather than abstaining around it. Before paying for a LoRA, I wanted to test a cheaper hypothesis: Microsoft published `microsoft/rad-dino-maira-2` after the original `microsoft/rad-dino`, trained on a more diverse CXR corpus, and the 2025 literature suggested the diversity might close part of the OOD sensitivity gap on its own. If the backbone swap closed enough of the gap, the LoRA scope shrinks.

**The A/B design, written to be falsifiable.** Spec from the parent: download the backbone, verify drop-in dimensionality, re-extract features on a curated 123-image subset (50 OOF + 50 NIH + 23 Cohen blind) under each backbone with IDENTICAL preprocessing (M12 invariant — `_harmonize`, seg+crop+letterbox, 37x37→8x8 pool, TXRV unchanged), train a FRESH tiny `TBHeadT2` per backbone under matched recipe (SEED=0, 80/20 stratified split, fusion+zonal+box levers), evaluate on the held-out 23 Cohen images. **GO** if maira-2 rescues ≥3 of the 4 M24 confident misses at `tb_prob ≥ 0.20` AND doesn't regress NIH No_Finding specificity by more than 2 cases AND a SEED=1 retraining doesn't flip more than 2/23 verdicts on the maira-2 arm. **NO-GO** if any of those three rails fail. **WATCH** if rescue is 1-2 of 4.

**Step 1 — dimensional sanity.** Both backbones output 768-d CLS + 1369 patch tokens (37x37 grid, patch_size=14, image_size=518). Drop-in compatible. But the feature SPACES are materially different on the same image: maira-2's CLS norm is 3.83× larger than base's, CLS cosine similarity is 0.105 on `public/samples/tb-sample-1.jpg`, mean per-patch cosine after the 8x8 pool is 0.484. This already told me the deployed `tb_head_t2.pt` — whose `LayerNorm` + `Linear(d_fuse, 128)` was fit to base-Rad-DINO features — cannot be run on maira-2 features without retraining. The A/B has to be matched-recipe.

**Step 2 — matched-recipe feature extraction.** `training/m27b/extract_features_maira2_ab.py` extracts features under each backbone on the same 123-image manifest (deterministic via SEED=0): 50 OOF rows sampled stratified by source (montgomery 12 / shenzhen 12 / qatar 13 / tbx11k 13, half-positive per source) + 50 NIH stress rows (12 No_Finding + 8 Fibrosis + 8 Nodule + 8 Consolidation + 7 Mass + 7 Effusion, prioritizing single-finding rows for cleaner per-mimic stress) + 23 Cohen blind. 42 seconds per backbone on warm M4/MPS. Row order is bit-identical across the two npzs so the head-to-head training sees exactly the same images, only the backbone differs. `data/features_maira2_ab.npz` and `data/features_base_ab.npz` (gitignored under `data/`).

## Milestone P0 — the locked-protocol baseline, and the bug in my own protocol (2026-05-25)

A GPT steelman of the planned backbone-swap milestone caught that re-fitting calibration per config on the evaluation surface IS test-set leakage. The fix had to be structural, so I built `training/locked_protocol.py`: `make_calibration_split(seed=7, cal_frac=0.20)` deterministically splits the 13,092-row LODO OOF into a 20% calibration slice (n=2,619) and an 80% evaluation slice (n=10,473), stratified by label×source. Plus `training/tta.py` — CXR-safe K=5 test-time augmentation (identity + H-flip + brighten/darken ±0.10 + contrast 1.10; no rotation, no crop) — wired into `TriageEngine` as opt-in `use_tta` + `use_locked_protocol`.

**Then my own locked protocol had a bug, and I nearly shipped it.** The first implementation re-fit the temperature from scratch by NLL on the calibration slice — which landed at T=2.85 — and then took the 95%-sensitivity threshold on those tempered scores, which fell to 0.098. The implementing agent noticed the surprising numbers and wrote an eloquent narrative defending them as "the honest fitted value, reported as-is." That narrative was wrong. Temperature and threshold are *coupled* (they define the same ROC point), so re-fitting T buys nothing for the decision — but it moves the probability space, and the entire M26 rule stack (`borderline_low=0.20`, sequelae escalator `0.7126`, asymmetric-evidence `0.88`) was derived in the *deployed* T=1.5915 space. The result: `borderline_low (0.20) > thr (0.098)` — an **inverted ABSTAIN band**. `abstain_rate` came out 0.0 not because no case was borderline, but because the band was empty. The whole safety net was silently dead, and the LODO eval-slice "spec" had collapsed to 0.645.

I caught it on a session-restart review by sanity-checking the one number that looked off (thr=0.098 vs the deployed 0.6105). The fix: the locked protocol locks the **deployed operating point** — T and thr read verbatim from `tb_threshold_t2.json` (1.5915 / 0.6105, the validated M22 values the M26 stack assumes) — plus the frozen seed=7 split. An assertion (`borderline_low < thr`) now makes the inverted band un-shippable: if anyone re-introduces a from-scratch re-fit that drifts the threshold below the borderline floor, `fit_locked_calibration` raises.

**Corrected frozen baseline** (deployed operating point, K=5 TTA, locked seed=7 eval slice — the reference every P1+ change measures against):

| Metric | Cohen blind (n=23) | LODO 80% eval slice (n=10,473) |
|--------|---|---|
| Strict sensitivity | 0.600 [0.25–0.85] | 0.846 |
| Strict specificity | — | 0.909 |
| Effective sens (with ABSTAIN rescue) | 0.636 | — |
| ABSTAIN band | valid [0.20, 0.6105) | valid |

Cohen strict sens 0.600 (TTA) is a real bump over M24's raw 0.455 — the K=5 averaging rescued a couple of borderline cases. The LODO eval slice (0.846 / 0.909) matches the in-distribution expectation, and the ABSTAIN band now fires.

**What I learned.** The steelman protected me from leakage in the *backbone-swap* plan, but the leakage-prevention machinery I built in response had its own subtler bug — a re-fit that silently disabled the safety net by drifting the probability space out from under thresholds that were calibrated elsewhere. The lesson compounds the original one: rigor isn't just "lock the threshold," it's "lock it to the operating point the rest of the system was built around, and add an invariant assertion so the lock can't quietly invert." The cheapest defense was the one number that didn't pass the smell test — 0.098 where 0.61 was expected — and the discipline to stop and check it instead of accepting the agent's confident narrative.

## Milestone P0.5 — the dataset I almost trained on is the holdout I most needed (2026-05-25)

The original P0.5 plan said to add the Mendeley Kiran/Jabeen Pakistani TB CXR set (2,494 TB+ + 514 normal) to the training corpus as multi-source diversity. Then I ran the first true external blind eval *on that exact set* — and it told a story that flipped the plan. That is the single most important number the project has produced, because it is the first measurement on a site the model never saw, and it is a *failure mode* (over-flagging new-site normals) distinct from the M24 atypical-TB sensitivity gap.

If I fold that set into training, I lose it. It is the only well-powered external TB+ cohort I have (2,494 positives — enough to satisfy the project's own "no ≥90% sensitivity claim without ~150+ held-out TB positives" rule with room to spare). So the corrected P0.5 makes two moves:

1. **Mendeley PK becomes the external validation holdout, structurally.** `build_index.py` now routes its 3,008 rows to a *separate* manifest, `data/index_external_holdout.csv`, tagged `split='external_holdout'`, and the training-index scan loop skips the `Kiran:Jabeen` raw directory entirely. The training `data/index.csv` row count is **unchanged at 13,260** — I verified the before and after are identical. The holdout cannot enter training by accident because it is never written to the file training reads.

2. **The lever against the specificity drift is negative-class diversity, not more positives.** The eval localized the drift to normals (US 0.984 vs Pakistani 0.675), so the immediately-available fix is to show the model more diverse normals during training. I already had 5,788 NIH ChestX-ray14 `No_Finding` films extracted through the same Rad-DINO + TXRV pipeline (`data/features_nih14.npz`). ## Milestone P1 — the surgical fix that wasn't, and the ablation ladder that said so (2026-05-25)

P1 was supposed to be the satisfying part: the M24 diagnostic had blamed `ZonalSoftOR`'s hard zone priors for attenuating mid-lung consolidative TB, the literature's preferred fix is a learned non-zonal attention pool, so I'd swap `ZonalSoftOR → SoftAttnPool`, add MixStyle for domain generalization, retrain, and watch the external numbers improve. Instead the ladder told me the head swap *hurts*, and the only lever that moved specificity is the one I almost didn't bother to isolate. That's a more useful outcome than a win would have been, and it only reads as honest because the locked protocol + single-lever discipline made it impossible to fudge.

I built four configs, each adding exactly one lever, **all evaluated under the FROZEN P0 locked protocol (T=1.5915, thr=0.6105, borderline_low=0.20 — never re-fit on any eval surface)** on the SAME cached eval inputs: (A) the deployed `zonal-softor` head on the original 4 sources; (B) +NIH `No_Finding` negatives in training; (C) +`SoftAttnPool` head swap; (D) +MixStyle (full P1). The prerequisite was extracting the held-out Pakistani cohort (3,008 images: 2,494 TB+ / 514 normal) once through the *identical* frozen `extract_features.py` pipeline into `data/features_mendeley_pk.npz` — a frozen-backbone read, so the cohort still never enters training, but every head variant now scores the exact same cached Pakistani features for a clean paired comparison. The LODO eval surface stayed fixed too: OOF logits over the 4 original sources only (NIH enters as training negatives, never as an eval row), and the deterministic seed=7 20/80 split carves the same 80% eval slice for every rung.

**What the levers actually did.** (1) **Negative diversity (B) is the only specificity lever that works** — adding 5,788 diverse US normals lifted Pakistani specificity at the frozen threshold from 0.203 to 0.449 (+0.246, paired CI +0.117–+0.198). But it cost LODO sensitivity badly: 0.880 → **0.714**, well below the 0.846 floor the gate protects. Showing the model more normals pushed the whole score distribution down, so at the unchanged threshold the in-distribution positives started falling under it. (2) **The SoftAttnPool head swap (C) is harmful, not neutral** — Pakistani AUROC dropped to 0.624 (paired −0.13), and specificity at the frozen thr collapsed to 0.043. The architectural fix the M24 diagnosis predicted made external *discrimination* worse, not better. (3) **MixStyle (D) partially rescues C** — full-P1 Pakistani AUROC of 0.776 is the ladder's best external ranking (paired +0.008–+0.034 vs A), and ECE is the lowest at 0.121, but MixStyle re-centred the scores upward, undoing B's specificity gain (PK spec back to 0.217), and LODO sensitivity stayed depressed at 0.781.

**The attention diagnostic confirmed the M24 mechanism — and that the mechanism wasn't the bottleneck.** On the 396 Pakistani TB+ that config D missed (the M24-class atypical cases), `SoftAttnPool`'s attention centre-of-mass row averaged **4.01 on the 0(apex)–7(base) axis** — i.e. it attended mid/lower lung, exactly where `ZonalSoftOR`'s +0.4 upper-zone prior would have under-weighted. So the head *did* learn to look where the consolidation lives, just as the diagnosis predicted. It just didn't translate into better external AUROC. The architectural story was directionally right at the attention level and wrong at the metric level.

**One honest caveat on the absolute numbers.** Config A is a *reproduction of the deployed architecture* (head retrained from scratch, Pakistani features scored without TTA, the M26 ABSTAIN band carved out), not a re-score of the exact shipped `tb_head_t2.pt` weights — so its Pakistani spec 0.203 looks far worse than the M0-era external-eval headline (spec 0.675, which used the actual shipped weights + TTA via `eval_external.py`). The two aren't directly comparable. What *is* trustworthy is the within-ladder paired comparison: same eval surface, same frozen protocol, one lever per rung, resampled-index delta CIs.

**A second, sharper caveat I added on review — and it downgrades the NIH lever.** Even the within-ladder *specificity-at-the-frozen-threshold* deltas are partly confounded: every config is a fresh retrain with its own probability scale, but all four are scored at the *deployed* model's threshold (0.6105). A model can move its specificity at a fixed threshold purely by re-centring its score distribution, with no change in discrimination at all. So spec-at-0.6105 conflates "the lever improved the model" with "the lever shifted the scores under a threshold that was never fit for this model." The two genuinely scale-fair signals are **AUROC** (rank-based, calibration-invariant) and **per-config recalibrated specificity at 95% sensitivity** (re-fit *only* the threshold, on a disjoint Pakistani slice, T held at the locked value). Under that lens the ladder reads: A 0.754 / recal-spec 0.027, B 0.764 / 0.082, C 0.624 / 0.030, D 0.776 / 0.091. The headline NIH "specificity win" (0.203 → 0.449 at the frozen thr) **largely evaporates** when each model is allowed its own threshold (recal-spec 0.027 → 0.082, still near-zero), and B's AUROC edge over A (+0.010) sits inside the noise (CIs overlap). What survives *both* lenses, robustly: the SoftAttnPool head swap (C) hurts discrimination (AUROC 0.764 → 0.624, non-overlapping CIs), and **no rung clears the gate**. I'm reporting all three views — the absolute floor, the frozen-thr relative signal, and the scale-fair signal — without smoothing over the discontinuity; the NO-GO holds under the strictest of them.

**Verdict: NO-GO on shipping the P1 head; negative diversity is marginal-not-transformative (a weak WATCH, not a lever).** Per the GO gate — improve Pakistani specificity *without* dropping LODO sensitivity below 0.846 or Pakistani sensitivity below 0.753 — **no config passes.** B improves PK spec *at the frozen threshold* but tanks LODO sens to 0.714, and under the scale-fair lens (above) its real effect is +0.010 AUROC inside the noise; C regresses on every axis; D barely touches PK spec and still sits at LODO sens 0.781. **Three drift tripwires fired** (§B worst/eval-fold sensitivity drop): LODO eval-slice sensitivity fell below 0.846 in B (0.714), C (0.716), and D (0.781). Per the discipline I stopped and recorded rather than trusting a flattering external AUROC. I am NOT deciding the gate myself — this goes back to the coordinator with the honest table.

**What I learned.** The locked protocol earned its keep here. A pre-P0 version of me would have run the full P1 stack (head + MixStyle + NIH all at once), seen the external AUROC tick up to 0.776, and called it a win — never noticing that the LODO sensitivity quietly cratered or that the head swap alone was actively harmful. The single-lever ladder turned "P1 got better" into "negative diversity is your specificity lever, the attention head is a liability, MixStyle is a calibration band-aid, and you can't get the spec without paying sensitivity at this operating point." That last clause is the real finding: at the image-only external ceiling (~67–70% spec at 90% sens per the field literature), the threshold and the training mix are coupled, and you cannot tune one rung of the ladder to beat the gate without re-paying on another. The next move isn't another architectural lever — it's either per-site recalibration (a different flow, not a baseline change) or genuinely new TB+ diversity (PadChest/VinDr when the DUAs land), not a cleverer pooling layer.

### P2 — the backbone spike that a cheap probe stopped at the gate (`feat/p2-backbone-spike`)

**What I set out to do.** P1 localized the external-generalization bottleneck to "the backbone representation and/or data diversity, not the pooling layer." P2 was supposed to test the *backbone* half of that by swapping rad-dino for `microsoft/rad-dino-maira-2` (a drop-in, same-geometry DINO-family backbone already in my local cache) under a frozen-feature, fixed-head protocol. But extracting maira-2 over 13,092 training images plus the 3,008-image Pakistani holdout is hours of compute, so I gated it behind the cheapest possible discriminator first (**P2.0**): if a linear probe on the *already-cached* rad-dino features can't even match the deployed head's external 0.781, the representation caps there and a backbone swap is well-motivated; if it scores much higher, the lever is the head, not the backbone.

**The pre-registered gate, and where it broke.** I trained L2-logistic probes on three branches of the cached features — rad-dino CLS (768-d), TXRV (1042-d), and the fused concat — on the 4-source corpus, evaluated on the held-out Pakistani cohort under patient-level bootstrap (pAUC at FPR≤0.2 and sensitivity at spec=0.70 alongside AUROC, because specificity is the failure mode). All three gate branches **collapsed externally**: Pakistani AUROC 0.604 (CLS), 0.599 (TXRV), 0.605 (fused), every CI well below the deployed head's 0.781. By the literal rule that says PROCEED (the probe is *not* ≥+0.05 above 0.781), maira-2 was greenlit.

Then I checked the premise. The deployed head doesn't read the CLS token at all — it pools the 64 *spatial* patch tokens through a zonal-soft-OR (an OR/max-style readout) with an apical prior and box supervision. A CLS probe is therefore the **wrong discriminator** for this head's signal path. So I added post-hoc patch-token probes (flagged in the JSON as hypothesis-generating, since they consult the Pakistani set): patch-mean PK AUROC 0.593, but **patch-MAX PK AUROC 0.853 [0.838, 0.868]** — *above* the deployed head's 0.781, with CIs cleanly apart. A simple max-over-tokens linear readout on the *unchanged* rad-dino features extracts more external TB signal than the trained head does.

**The honest call: INCONCLUSIVE, not PROCEED.** The literal CLS gate said PROCEED, but it was a misspecified discriminator. The patch-MAX result falsifies the naive rationale for the swap: rad-dino's representation does hold ~deployed-level (and better) external signal, but only in its spatial tokens accessed via an OR-style pool. That points the lever at **pooling/head/training**, not the backbone *family* — and maira-2 is the same DINO family, so it's the lower-priority move. Per the discipline, I did **not** spend the hours extracting maira-2; I committed P2.0, recorded the finding, and surfaced it for a strategy decision. (I noted the multiple-testing caveat: patch-MAX is post-hoc and consulted the Pakistani set, so 0.853 is hypothesis-generating, not a clean test; it also doesn't satisfy the original +0.05 STOP margin as a *preregistered* stop. It is, though, decisive enough to make the backbone swap premature.)

**One tripwire I had to defuse honestly.** The probe's same-corpus "LODO-eval" slice scores 0.98–0.99 AUROC — a +0.07 jump over the deployed head's true-LODO 0.923, which would trip the §B "LODO AUROC jump >+0.05" tripwire. But it's not a real jump: that slice is a deterministic seed=7 *same-site* holdout (both the probe-train and eval slices contain all four sources), an in-distribution reference, not leave-one-*dataset*-out. The 0.99 must not be read as beating the held-out-site baseline; I labelled the caveat in the JSON meta and the log so the number can't mislead later.

**What I learned.** Pre-registering a gate doesn't help if the gate's measurement instrument is wrong for the system under test. The CLS probe answered "does the CLS token carry external TB signal" (no), when the question was "does the *representation* carry it under the head's actual readout" (yes, via patch-MAX). The cheap discriminator still earned its keep — it cost minutes, not hours, and it redirected P2 from "extract maira-2" to "the pooling/head readout is where the external signal is being left on the table." The framing stays narrow: this says *under a frozen-feature linear-readout protocol, an OR-style spatial pool recovers more external rad-dino signal than the deployed head* — not "the backbone is irrelevant." The recommendation back to the coordinator: treat maira-2 as lower-priority, prioritize the pooling/head readout (and confirm any winner on an untouched external site, since the Pakistani set is now a selection surface), and keep CheXFound (a different family/scale) as the conditional follow-up if a head-side investigation also stalls.

#### P2.0b — putting the 0.853 on trial (is patch-MAX real TB signal or an artifact shortcut?)

**Why I distrusted my own number.** The 0.853 was *selected by consulting the Pakistani set*, and channel-wise MAX over 64 DINO patch tokens is exactly the readout most prone to a shortcut: a single high-norm patch from burned-in text, an AP/PA or side marker, a collimation edge, or an exposure cue can dominate the max, and DINO patch features encode all of those. Before I let "patch-MAX recovers external signal" stand as a finding, I ran a five-check artifact-discrimination battery on the *already-cached* features (no re-extraction). I fixed the probe to the **same estimator that produced 0.853** — the seed=7 cal-slice fit, n=2619 — and confirmed it reproduces 0.853 to four decimals before any check ran. (A side note that already smelled of fragility: the *same* max probe trained on the full source corpus instead of the 2619-row cal slice falls to 0.812. Part of the 0.853 rode a small fit sample.)

**The decisive clean test — LODO-replay, no Pakistani at all.** Leave-one-*source*-dataset-out: train the max probe on 3 source datasets, test on the held-out 4th, rotate over all four. If "max generalizes better" were a general property, max should beat mean and CLS on held-out *sources* too. It doesn't — it's the **worst** pool. Patient-weighted mean AUROC: **max 0.822, mean 0.868, CLS 0.908.** On clean held-out sources the CLS token I'd written off in the gate is the *strongest* readout and raw-max is the weakest. So the max-pool's edge is *Pakistani-specific*, not a general "max transfers better" effect — which is the signature of a selection/shortcut artifact, not transferable signal.

**But the other four checks complicate the artifact story.** (1) *Channel ablation* — drop the top 1/5/10 max channels by |coef|, refit, re-eval PK: it degrades *gracefully* (drop-top-5 actually nudges PK AUROC up to 0.871), so the lift is **distributed across channels, not driven by a handful of outlier dimensions.** (2) *Robust pools* — replace raw max with 95th-percentile, top-8-mean, low-temperature log-sum-exp, and winsorized max (all suppress the single hottest token): they **hold at 0.81–0.86**, while plain MEAN collapses to 0.593. So the signal needs an OR/max-style readout, but it is **not a lone-hot-patch artifact** — robustified maxes keep it. (3) *Token L2-normalization before pooling* — kill each token's magnitude (exposure/quality/contrast cues) and keep only direction, then max: 0.853 → **0.828, survives.** That argues the lift is *directional/semantic*, not a brightness/exposure shortcut. (4) *Negative control — predict SITE not TB*: the max readout predicts source dataset at 0.95 accuracy (chance 0.62) and Pakistani-vs-source at AUROC **1.000.** Site is near-perfectly legible from these features — a high shortcut prior — but that's expected for any DINO readout and doesn't on its own kill the TB claim.

**The honest verdict: MIXED, and I'm retracting the strong reading.** The *specific* claim "patch-MAX is a generally better readout that recovers transferable TB signal" does **not** survive — check 1 kills it cleanly (max loses to mean and CLS on held-out sources). But the crude "it's just a single hot artifact patch" reading doesn't survive either: the lift is distributed (check 2), robust to outlier-suppressing pools (check 3), and largely direction-driven rather than magnitude-driven (check 4). The fair conclusion is that the 0.853 is a **Pakistani-specific / selection effect inflated by a small fit sample**, layered on top of *some* directional rad-dino signal that no readout I tried converts into a clean held-out-*source* win. What this does to the P2 narrative: it **weakens** the earlier "maira-2 is lower-priority because rad-dino already holds untapped external signal in its spatial tokens" inference — that inference leaned on the 0.853 being real transferable signal, and the LODO-replay says it isn't a general property. The cleanest reading now is that *CLS-on-clean-LODO (0.908)* is the strongest evidence that rad-dino holds external signal, accessed through a *better-trained head*, not through a max pool. I'm letting checks 1/3/4 decide and **not** overclaiming in either direction; the next-step decision is the coordinator's.

**One precision I have to add, because it changes the conclusion.** "CLS-on-clean-LODO (0.908) → a better-trained head is the answer" reads better than the evidence supports: that 0.908 is *cross-source* generalization **within** my four training datasets, and the CLS probe **collapses to 0.604 on Pakistani**. Source-LODO is not external. Lining up everything I now have on the actual external (Pakistani) cohort — CLS 0.604, mean 0.593, raw-max 0.812 (0.853 on the small fit sample, non-replicating), the deployed zonal-soft-OR head 0.781, the P1 soft-attention head 0.624 — every frozen-rad-dino-feature readout I have tried clusters at **0.60–0.81 externally while sitting at 0.90+ cross-source**. The deployed head (0.78) is already at the top of that external band; there is no free readout lift hiding in these features. That is the convergent finding of three rigorous phases: **P1 (a learned pooling head), P2 (a different backbone), and P2.0b (a max/OR readout) all fail to close the Pakistani gap, and they fail the same way** — image-only model surgery on frozen features circles a ~0.78–0.81 external ceiling. The gap is domain shift, not a representation or readout deficiency I can engineer around. The honest levers left are the ones that change the *data* the model sees or the *process* around it: geographic TB-positive diversity in training (PadChest/VinDr when the DUAs land, or a held-out Pakistani training split), per-site recalibration at deployment, and multimodal fusion (CRP + CXR) for any WHO-grade ambition. CheXFound (a different family/scale, not just a different DINO) remains the *only* image-only architecture bet still standing, and it is now a low-prior one.
